# 海豚社区 App 未完成事项推进规划

**作者：Manus AI**

## 一、总体判断

当前剩余未完成项并不适合并行发散推进，而应该按照“**先止错，再防重，再对齐线上，再补生态归因**”的顺序收口。原因很明确：现在主路径已经具备基础可用性，但距离真实上线还差最后一层生产约束。如果顺序错了，例如先去做 Builder Code，或者先继续扩 token 支持，反而会把问题从局部复杂化扩散到全链路。

因此，后续规划我建议分成 **四个连续阶段**，每个阶段都有明确产出、验收口径和进入下一阶段的前置条件。

## 二、四阶段推进顺序

| 阶段 | 优先级 | 目标 | 预计产出 |
|---|---|---|---|
| Phase A | P0 | 补齐执行前风控 | 金额上限、白名单链、异常滑点拦截 |
| Phase B | P0 | 补齐幂等与防重复提交 | idempotency key、时间窗口、防重复执行记录 |
| Phase C | P0 | 对齐 serverless 线上入口 | `api/onchain/index.ts` 与 Express 逻辑一致 |
| Phase D | P1/P2 | 完成生态归因与发送层闭环 | Builder Code 注入、X Layer 真实发送层评估 |

## 三、每一阶段具体怎么做

### Phase A：先补风控规则

这一阶段是最紧急的，因为它决定了系统能否在用户误输、AI 误解、行情异常波动时及时止损。建议直接在 `execute` 路由做服务端强约束，而不是只靠前端提示。

| 子任务 | 具体动作 | 验收标准 |
|---|---|---|
| 金额上限 | 在 `server/_core/onchain-os-routes.ts` 和 `api/onchain/index.ts` 中统一加入单笔 `$10,000` 限制 | 超限请求直接返回 400，错误文案清晰 |
| 白名单链 | 仅允许 ETH / BSC / Solana / X Layer | 非白名单链请求被拒绝 |
| 异常滑点拦截 | `slippagePercent > 5` 直接拒绝 | 高滑点请求不进入执行层 |
| 错误分类 | 区分参数错误、风控拦截、执行失败 | 日志中能看出失败原因类型 |

这一阶段完成后，系统至少具备“**不该执行的交易先拦掉**”的能力。它是后面所有步骤的基础，因为没有风控，幂等和轮询都只是对错误交易做更稳健的处理。

### Phase B：再补幂等与防重复提交

当风控完成后，第二个最危险的问题就是重复执行。现在已经有 `onchain-tx-store`，所以非常适合直接把幂等做在服务端执行入口。

| 子任务 | 具体动作 | 验收标准 |
|---|---|---|
| 幂等 key 设计 | 基于 `userId + chainIndex + amount + fromToken + toToken + 时间窗口` 生成 key | 相同请求在窗口内得到同一任务结果或明确拒绝 |
| store 查询 | 在执行前查询 `idempotencyKey` 是否已存在活跃任务 | 命中时不重复下发执行 |
| 状态策略 | 对 `awaiting_confirmation / executing` 判定为活跃任务 | 活跃任务不可重复广播 |
| 日志记录 | 幂等命中写入 `onchain-tx-store` 日志 | 审计中可追踪重复点击与重复请求 |

这一阶段完成后，系统会从“能执行”升级到“**不会因为重试或多次点击而重复执行**”。

### Phase C：把 serverless 入口补到与本地一致

当前最大的工程风险之一，是本地 Express 路由已经前进了一步，但线上 `api/onchain/index.ts` 还没有完全同步。上线前必须把两边逻辑做成同一套规则，否则线上联调会出现“本地没问题，线上行为不一致”的情况。

| 子任务 | 具体动作 | 验收标准 |
|---|---|---|
| 执行风控对齐 | 把 Phase A 的规则同步到 serverless | 本地与线上同参同结果 |
| 幂等逻辑对齐 | 把 Phase B 的 key 与 store 逻辑同步到 serverless | 重复请求在线上也能拦截 |
| 返回结构对齐 | 保证两边都返回统一 `phase`、`txId`、错误结构 | 客户端无需区分来源 |
| 联调验证 | 本地与线上分别跑正常、超限、高滑点、重复提交四类用例 | 四类结果一致 |

这一阶段完成后，才可以说“**不是只在本地可用，而是部署后也稳定**”。

### Phase D：最后处理 Builder Code 与真实发送层

Builder Code 的优先级不应该早于前三项，因为它不解决安全与稳定问题，而是解决归因与生态合作问题。正确顺序是先把主链路做稳，再去接入真实发送层。

| 子任务 | 具体动作 | 验收标准 |
|---|---|---|
| 发送层评估 | 判断是否引入 `viem/wagmi` 或其他 X Layer 发送层 | 形成技术决策文档 |
| 注入位点落地 | 在 `sendTransaction` 层注入 `dataSuffix` | 代码中有明确实现点 |
| Builder Code 验证 | 用测试交易确认归因参数已携带 | 交易链路参数检查通过 |
| 风险回归 | 确认 Builder Code 注入不影响现有 phase/receipt 流程 | `pnpm check` 与联调通过 |

## 四、建议的执行节奏

如果按你现在的项目节奏，我建议不要把剩余事项拆得太碎，而是按下面这个节奏推进。

| 节奏 | 工作内容 | 目标 |
|---|---|---|
| 第 1 轮 | Phase A + Phase B | 先把“错误执行”和“重复执行”风险降下来 |
| 第 2 轮 | Phase C | 让线上入口与本地一致，完成部署前闭环 |
| 第 3 轮 | Phase D | 处理 Builder Code 和生态归因 |

这意味着，**下一轮开发我最应该直接开干的，不是新功能，而是风控、幂等、serverless 对齐。**

为了方便我后续持续自主推进，我建议把剩余事项重写成下面这组可执行任务。

| 任务 ID | 优先级 | 事项 | 完成定义 |
|---|---|---|---|

## 六、最终建议

如果只用一句话概括剩余规划，那就是：

> **先把交易系统做成“不会错、不会重、线上一致”，再去做“归因更完整、生态能力更强”。**

这条顺序是最符合你现在项目状态的。因为从产品上线风险看，真正致命的问题不是少了 Builder Code，而是风控缺失、重复执行和线上行为分叉。

所以，接下来我建议我就按这个顺序继续推进：**先 Phase A、再 Phase B、然后 Phase C，最后才是 Phase D。**
