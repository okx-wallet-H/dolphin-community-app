# 海豚社区 App 下一阶段完成内容规划

**作者：Manus AI**

## 一、当前阶段判断

当前仓库已经完成了 Onchain 主路径的第一轮生产化收口，重点包括统一交易 phase 状态机、交易任务持久化骨架、受控回执轮询、执行风控、幂等防重复提交，以及 Express 与 serverless 执行入口对齐。也就是说，系统已经从“能跑通的链上演示链路”进入了“具备上线前控制面的真实交易链路”。

接下来要完成的内容，不应该再平均分散推进，而应该围绕 **真实发送层闭环、用户可感知的交易主线程体验、跨页面联动、恢复能力与发布稳定性** 五个方向连续收口。当前最核心的原则是：**先把主交易链路做得更真实、更稳定，再去扩展外围页面和体验层联动。**

## 二、接下来建议的整体顺序

| 阶段 | 优先级 | 主题 | 目标 |
|---|---|---|---|
| Phase 1 | P0 | Builder Code 注入与真实发送层闭环 | 让 Builder Code 真正进入 X Layer 发送层，而不是只停留在配置与预留文档 |
| Phase 2 | P0 | 对话页交易主线程与确认卡片生产化收口 | 让用户在聊天页的确认、执行、失败、恢复体验完全闭环 |
| Phase 3 | P1 | 钱包页、交易页、赚币页与对话联动 | 让对话式交易能力从单页面扩展为全 App 的统一操作中心 |
| Phase 4 | P1 | 社区页与策略发现入口收口 | 让社区内容真正转化成可执行的链上动作与对话入口 |
| Phase 5 | P1/P2 | 恢复机制、稳定性、测试与发布机制 | 把系统从“具备主链路”推进到“可持续上线与维护” |

## 三、Phase 1：Builder Code 注入与真实发送层闭环

这一阶段仍然是当前的第一优先级。前面的风控、幂等和 serverless 对齐，解决的是“不要错”和“不要重”；而 Builder Code 与真实发送层，解决的是“交易是否真正进入目标生态发送路径，以及是否具备归因闭环”。

| 子任务 | 具体内容 | 完成定义 |
|---|---|---|
| TASK-0501 | 盘点当前仓库中所有潜在发送层位点 | 明确现有链上发送到底由谁执行，确认是否仍缺真实 `sendTransaction` 落点 |
| TASK-0502 | 设计 X Layer 发送层接入方案 | 明确是通过现有 SDK 承接、桥接 Agent Wallet，还是补独立发送层 |
| TASK-0503 | 在真实发送层注入 Builder Code / `dataSuffix` | 代码中出现明确注入点，且不再只停留在文档层 |
| TASK-0504 | 完成 Builder Code 验证 | 至少完成一次可验证的参数携带检查与链路确认 |

这一阶段的验收口径不是“代码里多了一个配置项”，而是：

> **能够明确证明 Builder Code 已经进入真实交易发送路径，并且不会破坏现有 phase、receipt 与风控逻辑。**

## 四、Phase 2：对话页交易主线程与确认卡片生产化收口

即便底层执行已经更稳，用户真正感知到的仍然是聊天页。如果主线程体验还不够完整，用户会把系统理解成“技术能力已接入，但产品体验仍然像测试版”。因此，这一阶段是与 Builder Code 并列的重要前台收口任务。

| 子任务 | 具体内容 | 完成定义 |
|---|---|---|
| TASK-0601 | 收口 swap 与 transfer 卡片文案和状态提示 | 所有关键动作文案统一为真实执行语义，不再出现旧兼容表达 |
| TASK-0602 | 完善失败态、超时态与恢复态 UI | 用户能清楚区分“待确认、执行中、失败、可重试、已完成” |
| TASK-0603 | 增加任务恢复入口 | 回到聊天页后能继续看到上一次交易的阶段与结果 |
| TASK-0604 | 统一风险提示与执行确认表达 | 金额上限、滑点、链路风险等提示前后一致 |

这一阶段完成后，用户看到的将不只是“能发起交易”，而是 **完整的可理解交易过程**。

## 五、Phase 3：钱包页、交易页、赚币页与对话联动

当前对话页已经开始成为真正的操作中心，但如果钱包页、交易页、赚币页依然是分散信息岛，那么 App 整体会出现能力割裂。下一步需要把“看见信息”和“执行动作”统一起来。

| 子任务 | 具体内容 | 完成定义 |
|---|---|---|
| TASK-0701 | 钱包页资产跳转到对话主线程 | 用户可从资产直接发起“卖出 / 兑换 / 转账 / 分析” |
| TASK-0702 | 交易页行情/币种跳转到对话执行 | 用户从行情看到机会后可直接进入对话交易 |
| TASK-0703 | 赚币页收益产品转对话承接 | 让赚币动作由对话提供解释、确认与执行承接 |
| TASK-0704 | 全局统一 token / address / chain 上下文 | 不同页面进入对话时上下文不丢失 |

这一阶段的目标是把 App 的多个页面都变成 **对话式执行入口的前置发现层**。

## 六、Phase 4：社区页与策略发现入口收口

社区页不应该只是内容展示区，而应该成为“策略发现 → 转入对话 → 执行动作”的前端漏斗。如果这里不收口，社区能力和链上执行能力之间仍然是断开的。

| 子任务 | 具体内容 | 完成定义 |
|---|---|---|
| TASK-0801 | 社区帖子/策略卡片增加对话转化入口 | 点击后能携带上下文进入聊天页 |
| TASK-0802 | 社区搜索与聊天搜索语义打通 | 搜索结果不再是孤立展示，而能转为执行建议 |
| TASK-0803 | 热门策略/代币卡片支持 AI 解读 | 用户能从内容直接进入“要不要做、怎么做”的对话承接 |

这一步完成后，社区页才会从“内容页”升级成“增长与转化页”。

## 七、Phase 5：恢复机制、稳定性、测试与发布机制

这是把项目从“主链路可用”推进到“能够稳定发布和运营”的关键阶段。前面虽然已经有 `onchain-tx-store` 和受控轮询，但还没有形成更完整的后台恢复与发布标准。

| 子任务 | 具体内容 | 完成定义 |
|---|---|---|
| TASK-0901 | 完善交易恢复机制 | retryCount、lastError、恢复策略、超时语义更完整 |
| TASK-0902 | 设计后台轮询或恢复扫描机制 | 不完全依赖前台请求触发状态恢复 |
| TASK-0903 | 建立关键用例测试矩阵 | 正常、超限、重复提交、失败恢复、链路中断等都有验证用例 |
| TASK-0904 | 建立发布前检查清单 | 环境变量、API 路由、主路径联调、截图审核、回归测试全部有记录 |

这一阶段的验收标准是：

> **即使用户中途离开、网络中断或请求重放，系统也能更稳地恢复、可查、可解释。**

## 八、我建议的实际推进节奏

从推进效率和风险控制来看，后面最适合按“三轮收口”来做，而不是一次性同时开很多线。

| 轮次 | 本轮重点 | 预期结果 |
|---|---|---|
| Round 1 | Phase 1 + Phase 2 | 完成真实发送层与聊天主线程闭环 |
| Round 2 | Phase 3 + Phase 4 | 完成多页面到对话的统一联动与转化闭环 |
| Round 3 | Phase 5 | 完成恢复、测试、发布与上线前稳定性收口 |

## 九、当前最建议我马上继续做什么

如果现在继续推进，我建议我下一步直接进入下面这个顺序：

| 顺序 | 下一动作 | 原因 |
|---|---|---|
| 1 | 先定位真实发送层与 Builder Code 注入位点 | 这是当前唯一还未真正落地的 P0 交易能力缺口 |
| 2 | 再补聊天页恢复态、失败态与确认卡片收口 | 这是用户最直接感知的产品完成度 |
| 3 | 然后开始钱包页/交易页/赚币页向对话联动 | 让整个 App 形成统一操作中心 |
| 4 | 最后推进后台恢复、测试和发布清单 | 为后续真机验证与上线做准备 |

## 十、最终结论

接下来要完成的内容，已经不是继续“补更多能力点”，而是把现在已有的链上能力 **做成真正完整的产品闭环**。因此，后续顺序不应该乱：

> **先做 Builder Code 与真实发送层，接着收口聊天页交易体验，再做多页面联动，最后补恢复与发布稳定性。**
