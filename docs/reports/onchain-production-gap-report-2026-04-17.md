# 海豚社区 App 代码检查与不足项报告（2026-04-17）

**作者：Manus AI**

## 一、执行结论

我已对当前新仓库 `/home/ubuntu/dolphin-community-app-devrepo` 的 App 代码进行持续检查，并围绕 **OKX Onchain OS / Agent Wallet 主路径** 做了针对性的生产化收敛。当前仓库已经明显优于最初的“骨架可跑但未形成生产闭环”的状态，尤其是在 **交易状态机统一、聊天主线程承接、交易任务持久化骨架、回执轮询稳健性** 这四个关键方向上已经完成一轮修复。

不过，从即将上线的标准来看，仓库依然存在一批必须继续收口的不足。这些不足不再是“能否演示”的问题，而是 **能否稳定、可恢复、可防错、可追责地承接真实交易** 的问题。换言之，当前代码已进入“可以继续向生产靠拢”的阶段，但还不能判断为“生产闭环完成”。

## 三、当前仓库仍然存在的不足

### 1. 交易风控仍未落地到执行入口

当前执行入口虽然已经有了持久化骨架，但 **生产级风控规则仍未加到 execute 路由与 serverless 入口**。这意味着单笔金额限制、白名单链控制、异常滑点拦截等关键保护仍未成为硬规则，而只是“下一步要做”的计划。

| 不足点 | 当前状态 | 风险 |
|---|---|---|
| 单笔金额上限 | 未实现 | 大额误操作无法在服务端前置拦截 |
| 白名单链控制 | 未实现 | 非目标链或未验证链路可能被误放行 |
| 异常滑点拦截 | 未实现 | 波动异常时可能提交高损耗交易 |
| 本地路由与 serverless 一致性 | 未同步 | 本地与线上行为可能分叉 |

### 2. 幂等与防重复提交仍是空缺

当前 `server/_core/onchain-tx-store.ts` 已经预留了 `idempotencyKey` 字段，但 **真正的幂等策略并未接入**。这意味着在聊天页重复点击、签名回跳重复触发、移动端网络重试、前端重发请求等场景下，仍存在重复执行的潜在风险。

这类问题在真实交易产品里往往比一般 CRUD 重试更敏感，因为它直接影响资产安全、订单重复、用户信任与后续客服成本。因此，这一项仍然属于 **P0.5~P1 之间的关键缺口**，不应继续延后。

### 3. 回执轮询虽已增强，但恢复链路还不完整

本轮已经把回执查询升级成受控轮询，但目前它仍然偏向“请求内轮询”，还没有完全演进到 **任务驱动的恢复模型**。也就是说，现在已经有了 `onchain-tx-store`，却还没有进一步形成：

> “任务创建 → 执行 → 轮询 → 写回 → 失败重试计数 → 恢复查询入口”的完整交易任务状态机。

| 维度 | 当前状态 | 仍缺什么 |
|---|---|---|
| 即时回执承接 | 已具备基础能力 | 还缺更明确的超时策略与最终失败语义 |
| 任务状态持久化 | 已有骨架 | 还缺系统化恢复任务扫描与重试策略 |
| retryCount | 字段已存在 | 尚未形成真正的递增与决策规则 |
| 错误记忆 | 已有 `lastError` | 尚未形成可查询、可运营的恢复机制 |

### 4. Express 侧能力已前进，但 Vercel serverless 入口仍未完全同步

本轮持久化接入的是 `server/_core/onchain-os-routes.ts`，但 `api/onchain/index.ts` 这一条线上入口仍未同步完成同等级别的风控、持久化和幂等逻辑。这会带来一个非常现实的问题：**本地联调通过，并不等于线上部署行为一致**。

对于即将上线的产品，这种“双入口能力不一致”是非常危险的。因为最终问题不会体现为“代码报错”，而会体现为 **相同用户行为在不同环境下得到不同结果**。

### 5. Builder Code 仍未注入真实发送层

Builder Code `yf83qce657mgxsjw` 已确认需要在 **X Layer 真实发送层的 `dataSuffix`** 中注入，但当前仓库仍然缺少明确的发送层落点，因此该能力仍处于“确认结论但未实现”的状态。

这项不足的直接影响并不是交易无法发生，而是 **归因与生态合作能力无法兑现**。如果项目后续依赖 Builder Code 做链上归因、合作统计或返佣结算，那么它必须在发送层真实落地，而不能长期停留在研究文档里。

### 6. transfer 链路仍然偏“卡片承接”，尚未达到与 swap 同等级生产闭环

目前 transfer 卡片已经统一进 phase 模型，且回跳后能承接部分执行状态，但从系统完整性上看，它仍未达到 swap 路径的成熟度。具体来说，transfer 侧目前更像是 **前端主线程承接增强**，还不是完整的 **服务端任务模型 + 风控 + 幂等 + 回执恢复** 闭环。

这意味着，如果产品上线后把“转账”也作为核心高频动作，那么 transfer 路径还需要单独做一轮对齐。

## 四、对当前代码质量的总体判断

当前仓库最值得肯定的地方，是方向已经收敛正确：**OKX 负责底层执行安全，海豚社区 App 负责意图理解、确认卡片、风控与状态承接。** 这条职责分工在代码结构上已经开始得到体现，尤其是在 `onchain-os.ts`、`chat.tsx`、`lib/_core/api.ts` 三层之间已经出现比较清晰的边界。

但是，真正决定是否能上线的，不是“主路径是否可演示”，而是下面三件事是否闭环：

| 上线判断项 | 当前判断 |
|---|---|
| 状态机是否统一 | **基本完成第一轮收敛** |
| 持久化与回执恢复是否具备基础闭环 | **已建立骨架，但尚未完全闭环** |
| 风控、幂等、部署一致性是否到位 | **尚未完成** |

因此，我对当前仓库的判断是：**方向正确，骨架正在成型，但仍未达到“可放心上线真实交易”的完成度。**

## 六、最终结论

如果站在“仓库代码检查并找出不足”的角度，当前最重要的结论不是发现了多少问题，而是已经非常明确地识别出：

> 当前仓库的最大不足，已经从“能力缺失”转移为“生产闭环未完成”。

也就是说，主路径方向已经正确，核心骨架也已搭好，但仍然需要继续把 **风控、幂等、恢复、部署一致性、Builder Code 注入** 这几项补完，才能从“能跑的 AI 交易体验”真正进入“可上线的 AI 交易产品”。

从推进顺序上，我建议下一轮直接以 **风控 + 幂等 + serverless 对齐** 为主，不再分散注意力。
