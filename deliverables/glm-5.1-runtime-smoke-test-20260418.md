# 海豚社区 App：GLM 5.1 真实联调与四大技能首批验证进展

**作者：Manus AI**  
**日期：2026-04-18**

本轮工作已经从“代码层接入完成”推进到“**真实模型可调用**”阶段。我已在项目本地联调环境中写入 GLM 5.1 运行配置，并使用官方兼容接口完成了一次最小化探测请求。探测请求成功返回 `200`，且模型实际返回了结构化 JSON 内容，这说明海豚社区当前聊天主链已经具备与 **GLM 5.1** 进行真实通信的基础条件。

在模型探测通过之后，我进一步复用了仓库中现成的聊天能力测试脚本，对四类中文自然语言输入进行了首批烟雾验证。这一轮验证覆盖了你定义的四大技能框架中的 **行情、赚币、钱包、交易** 四类代表性场景，用于确认模型是否已经能够把自然语言稳定映射到海豚社区现有技能编排链路。

| 测试项 | 用户输入 | 识别结果 | 执行状态 | 关键结论 |
|---|---|---:|---:|---|
| 行情 | 帮我查一下 ETH 最新价格 | `market` | `mockMode=false` | 已成功走到真实行情查询链路，并返回价格文本 |
| 赚币 | 帮我赚钱，给我一个稳健方案 | `earn` | `mockMode=false` | 已成功走到真实赚币方案链路，并生成产品建议 |
| 钱包 | 给我充值地址 | `deposit` | `mockMode=false` | 已成功走到真实 Agent Wallet 充值地址链路 |
| 交易 | 帮我用100U买ETH | `swap` | `mockMode=true` | 已正确识别为交易意图，但当前仍停留在交易确认/编排阶段，尚未完全切到真实执行态 |

从这轮联调结果看，**GLM 5.1 作为自然语言主交互模型已经可用**，而且在理解中文用户表达、归类四大技能、输出结构化意图方面表现稳定。当前最直接的成果不是“模型能聊天”，而是它已经能够作为海豚社区的 **意图中枢**，把用户自然语言路由到你定义的四个业务能力域中。

不过，本轮验证也明确暴露出一个当前最重要的阻塞点：**交易链路虽然能被模型准确识别，但仍未完全进入真实执行态**。换言之，模型侧的理解已经到位，交易侧的问题主要不在 LLM，而在交易确认卡片、报价预览、执行调用和真实广播之间的最后一段联通。这个问题与前面已经完成的钱包、行情、赚币真实接入形成了清晰对比，因此下一轮最应该优先补的是 **Swap 从 mock 编排切到真实执行**。

|---|---|---|
| GLM 5.1 官方接口可用 | 已完成 | 本地探测请求成功返回 200 |
| 聊天主链已读取真实模型配置 | 已完成 | 本地环境已接入真实模型地址、模型名与密钥 |
| 自然语言识别四大技能 | 已完成 | 四个代表性中文样例均被正确归类 |

下一步建议非常明确。首先，应继续沿着当前已经接好的聊天主链，优先检查 **Swap 预览、确认与执行** 的最后一段逻辑，把交易从“会识别、会生成确认语义”推进到“会真实报价、会真实执行”。其次，在交易链打通后，再补一轮面向终端页面的联调，让聊天页、钱包页和交易页共享同一套真实模型+真实 OKX 能力入口。这样做的好处是，后面的测试就可以进入你说的那种模式：**一个一个场景实际点测，而不是继续停留在架构推演层。**

| 建议测试顺序 | 目标 |
|---|---|
| 1. 继续打通 Swap 真实链路 | 让交易技能退出 mock 状态 |
| 2. 聊天页端到端联调 | 验证前端消息发送到模型、再到技能路由的闭环 |
| 3. 钱包与行情回归测试 | 确保新增模型联调未破坏已接好的真实能力 |
| 4. 赚币参数补全测试 | 验证不同用户说法能否稳定归到 earn 技能 |

本轮联调的原始输出文件已保存在本地测试产物中，可作为后续逐项回归的基线数据继续复用。整体上看，海豚社区已经完成了从“模型接入改造”到“真实模型开始工作”的关键跨越，接下来最值得投入的方向不是继续换模型，而是尽快把 **交易技能的最后一段真实执行链** 补齐。
