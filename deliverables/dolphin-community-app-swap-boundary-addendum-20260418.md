# 海豚社区 App：Swap 职责边界修订说明

**作者：Manus AI**  
**日期：2026-04-18**

## 一、边界修订结论

根据你的最新说明，海豚社区 App 在本项目中的职责边界需要重新界定为：**交易层面的 Agent 会自行与 Swap / OKX Skills 交互，App 不负责直接承接交易撮合、报价调用、广播上链或与 Swap 服务做一跳一跳的底层交互**。这意味着我上一版报告中把 `dex-swap.ts`、`onchain-os.ts` 一整条执行链当作“App 必须亲自接通的主路径”来审查，口径偏重了执行层本身，和你现在确认的产品职责边界并不完全一致。

在这个新边界下，海豚社区 App 的正确定位应当收敛为：它负责承接用户自然语言意图，生成结构化任务上下文，把任务交给 Agent，由 Agent 再去调用 OKX 已封装的 Swap / Skills 能力，并把结果、状态和错误回传给 App 展示与继续编排。[1] [2] [3] 因此，**App 端最重要的不是“自己把 Swap API 全打通”，而是“不要错误地接管本该由 Agent 负责的交易执行职责”**。

## 二、对上一版报告的修订

上一版报告里有一部分结论应继续保留，但也有一部分需要降级为“非主责范围的技术债”而不是“当前版本必须打通的缺陷”。

| 结论类别 | 是否保留 | 修订后的解释 |
|---|---|---|
| `mockMode` 被硬编码导致前端误报 | 保留 | 这仍然是 App 层问题，因为它会误导用户和调试判断 |
| Swap 意图解析入口不统一 | 保留 | 这属于 App/Agent 协同层问题，会影响把任务正确交给 Agent |
| 前端状态机与回执语义不统一 | 保留 | App 即便不直接执行交易，也必须正确承接 Agent 返回的状态 |
| `txHash`、`orderId` 等回执透传缺失 | 保留 | 这些字段仍属于 App 消费 Agent 返回结果时必须处理好的上下文 |
| `dex-swap.ts` 直接打 OKX DEX API 的完整执行链 | 降级 | 这不再是 App 主路径必须完成的能力，而更像遗留兼容代码或实验性接入 |
| 本地缺少 OKX 私有凭证导致真实联调未完成 | 降级 | 如果交易由 Agent 执行，则 App 本地未配置 DEX 私钥不再构成主阻塞 |

换句话说，上一版报告里最需要被纠正的一点，是我把“**App 是否已经亲自把 Swap API 跑通**”当成了成功标准；而在你现在明确的边界下，真正的成功标准应当改成“**App 是否已经把 Swap 这类交易意图正确交给 Agent，并把 Agent 的执行状态与结果稳定承接回来**”。

## 三、按新边界重新看，当前最值得关注的问题

在不要求 App 直接与 Swap 底层交互的前提下，我认为当前仓库最需要优先审查和改造的，是下面三类问题。

首先，是**任务边界是否干净**。如果仓库中仍然保留大量 `dex-swap.ts`、`onchain-os.ts` 这类“App 亲自执行交易”的代码，并且聊天主线程还会直接走这些路径，那么这会造成职责漂移：一部分交易由 Agent 做，一部分交易又由 App 自己试图做，最终系统会出现双路径、双状态机、双错误源。这个问题比“某个 API 能否打通”更严重，因为它会让整个平台架构长期不稳定。

其次，是**App 与 Agent 的协议是否稳定**。既然 Agent 会自己与 Swap/Skills 交互，那么 App 层就必须有一套明确的任务提交、执行中、待确认、成功、失败、补偿、重试、取消的状态模型，并且字段命名与展示语义要统一。也就是说，App 的关键不再是 `quote` 或 `swapTransaction` 的具体细节，而是像 `intent`、`taskId`、`phase`、`requiresUserConfirmation`、`resultSummary`、`txHash`、`errorCode` 这类面向编排和展示的上层协议。

最后，是**遗留直连能力是否会继续误导产品方向**。如果当前代码仍保留“直接请求 OKX DEX API、自己签名、自己广播、自己查回执”的大量实现，但产品设计上已经明确由 Agent 负责这些事，那么这些代码就不应再被视为主路径能力，而应当被标记为兼容层、实验层或待下线模块。否则后续开发会不断有人沿着错误方向继续补逻辑。

## 四、因此，后续整改建议也要改

在新的职责边界下，建议后续工作按下面的优先级推进。

| 优先级 | 建议 | 原因 |
|---|---|---|
| P0 | 明确并固化“App 不直接执行 Swap，Agent 才是交易执行主体”的架构说明 | 先把边界写清楚，才能避免继续误改主路径 |
| P0 | 统一聊天主线程到 Agent 的任务协议和状态字段 | 这是 App 真正要做好的核心主链 |
| P1 | 排查并收敛所有仍会让 App 直接打 Swap/DEX API 的入口 | 防止双路径并存 |
| P1 | 让前端卡片、消息、回调、回执都围绕 Agent 返回结果建模 | 避免 UI 继续绑定到底层 DEX 细节 |
| P2 | 将 `dex-swap.ts`、`onchain-os.ts` 等直连能力标记为兼容层或待下线模块 | 降低后续维护成本 |
| P2 | 仅在确有必要时保留少量只读查询或调试能力 | 保留排障价值，但不让其侵入主交易链 |

这也意味着，像“补真实 OKX DEX API 凭证并在本地把整条直连执行链测通”这件事，在新的边界下已经不是最高优先级了。除非你后续又决定让 App 重新承担交易执行职责，否则这类工作应被视为次要验证项，而不是当前版本的主目标。

## 五、修订后的总判断

按你现在确认的口径，海豚社区 App 的核心不是一个直接交易客户端，而是一个**面向 Agent 的任务承接与结果呈现前端**。因此，后续代码审查也应围绕这个定位展开：重点查的是 **意图是否被正确提交给 Agent、Agent 回来的状态是否被稳定消费、用户确认环节是否清晰、失败补偿与重试是否合理，以及遗留直连交易代码是否还在错误地充当主路径**。

所以，我对上一版报告的修订版一句话总结是：

> 不是“App 还没把 Swap 执行链打通”，而是“App 需要避免越权接管交易执行，并把自己收敛成 Agent 交易流程的编排与展示层”。
