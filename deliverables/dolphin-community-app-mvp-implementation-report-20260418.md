## 三、关键代码收口说明

### 1. 钱包页已经具备“资产列表 → 币种明细”主入口

此前 `wallet.tsx` 里的资产行只是静态展示，这与“点击 USDT 进入对应币种交易明细”的产品口径不一致。本轮已把资产行改造成可点击入口，并携带 `symbol`、`chainName`、`chainIndex`、`walletAddress`、`tokenPrice` 等必要上下文，使钱包页第一次具备了真正的 **账本承接分流入口**。

### 2. 新增币种详情页，开始承接账号明细职责

新建的 `app/token-detail/[symbol].tsx` 采用项目现有设计令牌实现了更正式的页面结构，顶部展示币种概览，中部展示交易记录列表。当前页面通过现有 `getDexSwapOrders()` 能力筛选与目标币种相关的真实记录，并将其归一为 `处理中 / 已完成 / 执行失败` 三种状态，满足你要求的“**成功和失败都要进账号明细**”这一最小 MVP 前提。

### 3. 赚币页已移除演示兜底

`earn.tsx` 中原本保留的 `fallbackStrategies` 演示型兜底已被删除，页面现在只会展示通过真实检索拿到的投资产品。如果当前没有真实产品，则页面直接进入空状态，不再出现演示卡片、演示策略或假数据暗示。这一点已经与“赚币页就是交易页，只有存在真实投资产品才展示”的产品口径对齐。

### 4. 聊天页交易卡片已从“重追单”转向“轻提示”

聊天页中的 SwapCard 与 TransferCard 已进行一轮正式化收口。本轮重点不是继续增强聊天主线程的状态机，而是**降低旧广播、续跑、回执追踪的视觉存在感**，把卡片表达调整为：

| 场景 | 当前卡片表达 |
|---|---|
| 兑换意图 | Agent 已接管兑换意图，展示核心报价字段与结果入口 |
| 转账意图 | Agent 已接管转账意图，展示核心地址字段与结果入口 |
| 主按钮 | 当需要签名时，引导交给 Agent Wallet 执行 |
| 次按钮 | 引导去钱包看明细，而不是继续在聊天页追单 |

也就是说，聊天页开始从“结果主线程”退回到“触发入口 + 去明细查看”的角色。

## 五、当前仍然存在的不足

虽然本轮已经把 MVP 关键路径落了下来，但从“偏正式产品化”角度看，仍有几处需要继续推进。

| 级别 | 不足点 | 说明 |
|---|---|---|
| P0 | 账号明细仍主要复用 DEX 订单数据 | 当前 `token-detail` 页面优先承接了 swap 类记录；若后续要完整覆盖转账、申购、赎回，需要统一到账本模型或服务端聚合接口 |
| P0 | 聊天页底层兼容逻辑仍在 | 本轮已收敛视觉和主路径文案，但旧的签名回调、续跑、轮询兼容代码仍存在于仓库中，后续应继续从架构层降级 |
| P1 | UI 设计师逐页审看尚未完成 | 按你的执行规范，钱包页、赚币页、聊天页、币种详情页仍需逐页过 UI 审核 |
| P1 | 尚未产出真机截图 | 本轮完成了代码落地与类型验证，但未在本轮交付中附上页面截图；若你要给 UI 设计师审看，下一步建议跑起预览并导出截图 |
| P1 | 结果回写仍需继续贯通 | 当前前端已具备明细承接页，但真实结果是否完整进入该页，还取决于后续统一记录写入链路 |
