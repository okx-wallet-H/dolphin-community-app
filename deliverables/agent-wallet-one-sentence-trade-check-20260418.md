# Agent Wallet 一句话交易能力复核（2026-04-18）

## 结论摘要

你这次纠正得对。老板清单里的 **“一句话交易”** 重点应该理解成：**不是固定测 `100U 买 ETH` 这个单例，而是要验证“任意金额、任意币种、自然语言一句话发起交易”的通用能力**。按这个口径重新核查后，当前结论是：

| 检查维度 | 当前结论 | 是否达标 |
|---|---|---:|
| 任意金额 | **部分支持**。只要语句能被当前解析器识别，金额本身不是写死的，`1`、`2.5`、`250`、`3.66` 这类数值都能解析出来 | 部分达标 |
| 任意币种 | **不达标**。解析层可以识别很多 symbol 文本，但真正进入聊天页执行时，币种解析仍受前端硬编码映射限制 | 不达标 |
| 自然语言一句话 | **不达标**。当前更偏向“换成 / 兑换 / swap to”句式；像“买”“卖”“用 X 买 Y”这类更自然的表达并未稳定覆盖 | 不达标 |
| OKX Skill 封装 | **已存在**。项目里确实已经封装了 `swap_quote` / `swap_execute` / `swap_approve_transaction` 等别名 | 达标 |
| 一句话交易主链路是否真正复用 OKX Skill 封装 | **未完全打通**。聊天页实际主链路仍然走 `onchain-os.ts -> dex-swap.ts`，没有完整复用统一的 OKX Skill 别名层 | 不达标 |
| 真实执行闭环 | **当前不可验证为真实成交**。本地环境缺少 DEX 实盘凭证，执行链路会自动回落到 mock | 不达标 |

因此，**117 这项目前仍然不能打勾**，但原因已经从之前的“100U 不够”修正为：**当前产品实现还没有达到“任意金额、任意币种、任意自然语言表达都能一句话交易”的通用能力标准**。

## 我重新核查到的关键事实

### 一、底层确实有 OKX Skill 封装，不是完全没有

项目里已经存在统一的 OKX Skill 友好别名：

| 能力 | 当前别名 |
|---|---|
| 获取兑换报价 | `swap_quote` |
| 构建兑换交易 | `swap_execute` |
| 获取授权交易 | `swap_approve_transaction` |
| 兼容旧命名 | `get_dex_quote` / `get_dex_swap` / `get_dex_approve_transaction` |

这说明你的判断是对的：**项目内确实已经有一层 OKX Skill 封装**，不是完全从零硬写的。

但是，聊天页的一句话交易主链路并没有完全走这层统一封装，而是直接桥接到：

> `onchain-os.ts -> dex-swap.ts`

而 `dex-swap.ts` 的真实执行又依赖本地环境变量：

- `OKX_DEX_API_KEY`
- `OKX_DEX_SECRET_KEY`
- `OKX_DEX_PASSPHRASE`
- 或它们的兼容别名

我实际核查当前环境，这些值现在都不存在。因此当前交易预览和执行都会回落到 **mock**。

## 二、“任意金额”并不是主要问题

我按多种金额做了解析测试，当前解析器对金额本身并没有写死：

| 测试语句 | 解析结果 |
|---|---|
| `把 1 USDT 换成 ETH` | 成功识别 |
| `swap 2.5 btc to eth` | 成功识别 |
| `把 250 PEPE 换成 USDT` | 成功识别 |
| `帮我把 3.66 sol 换成 usdt` | 成功识别 |
| `把 12 DOGE 兑换成 SOL` | 成功识别 |

这说明至少在**“换成 / 兑换 / swap to”** 这类句式里，当前实现对金额没有固定写死在 `100` 上。

所以你说 **“100 只是比喻”** 是对的；当前真正的问题不是金额写死，而是**自然语言覆盖范围**和**币种落地支持范围**。

## 三、“任意币种”现在只是解析层看起来像支持，执行层并不通用

我这次重新拆开看了两层：

### 1. 解析层

`parseSwapIntent` 的正则会接受形如：

- 数量
- `fromSymbol`
- `toSymbol`

所以像 `PEPE -> USDT`、`DOGE -> SOL` 这种文本，当前**解析层**是能识别出来的。

### 2. 执行层

但聊天页真正执行时，不是根据 symbol 去动态查币，而是使用硬编码代币映射：

| 链 | 当前硬编码支持 |
|---|---|
| EVM | `ETH`、`USDT`、`BTC`、`WBTC` |
| Solana | `SOL`、`USDT` |

也就是说，虽然 `PEPE`、`DOGE` 在解析阶段能被识别成字符串，但到了真正执行前的 `resolveSwapToken`，因为这些币不在硬编码表里，就会直接落入：

> “当前币种暂未接入”

因此，**当前 App 绝对还不能说支持“任意币种一句话交易”**。

## 四、“买 / 卖 / 用 X 买 Y”这类自然表达目前并未打通

我重点复测了你强调的自然语言交易口径。结果如下：

| 测试语句 | 当前结果 |
|---|---|
| `帮我用100U买ETH` | **无法识别**，返回 `unknown` |
| `买 0.2 BTC` | **无法识别**，返回 `unknown` |
| `把 1 USDT 换成 ETH` | 可识别 |
| `swap 2.5 btc to eth` | 可识别 |

这说明当前实现不是“任何一句自然语言都能交易”，而是：

> 只对较窄的一组换币句式有覆盖。

所以 **117 的真实失败点**，应该修正成：

- 不是只因为之前拿 `100U` 当例子；
- 而是因为 **“一句话交易” 的自然语言覆盖仍明显不完整**。

## 五、真实闭环目前还被 mock 回退挡住

我继续检查了交易预览与执行结果，当前状态是：

| 环节 | 当前结果 |
|---|---|
| Swap 预览 | `providerMode = mock` |
| 执行阶段 | `mockMode = true` |
| 执行回执 | 返回 mock success 样式，不是链上真实广播结果 |

这不是 OKX Skill 理论上不能做，而是当前这条主链路在本地环境里**没有实盘 DEX 凭证**，所以自动回退到了 mock。

因此，当前不能把这一项当作“已完成的真实闭环”。

## 六、对 117 的修正后判定

按照你修正后的标准，117 应该这样判断：

| 子标准 | 当前状态 |
|---|---|
| 任意金额 | 部分成立 |
| 任意币种 | 不成立 |
| 买/卖/用 X 买 Y 等自然语言 | 不成立 |
| 主链路真正复用统一 OKX Skill 封装 | 不完全成立 |
| 真实报价、真实执行、真实结果反馈闭环 | 不成立 |

因此：

- **117 仍然不能打钩**；
- 但我会把之前“因为 100U 不够所以不通过”的表述改掉，改成**更准确的产品能力判断**。
