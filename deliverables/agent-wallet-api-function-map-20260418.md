# Agent Wallet API 功能清单与海豚社区封装起点（2026-04-18）

作者：**Manus AI**  
日期：2026-04-18

## 一、结论先讲清楚

如果以你现在确定的产品前提——**“用户通过邮箱注册/验证，触发 Agent Wallet 生成，再围绕这个钱包去深度融合 OKX Onchain OS 生态”**——来倒推，第一步确实不是先讨论 Swap 页面怎么写，而是先把 **Agent Wallet API 到底负责什么** 划清楚。

从 OKX 官方资料看，**Agentic Wallet** 的定位不是普通查询接口，而是把 Agent 从“只会回答问题”推进到“能持有资产、签名并提交链上交易的执行器”。官方明确强调：用户可以通过邮箱快速创建钱包，私钥在 **TEE** 中生成与托管，交易在执行前会经过风险模拟与评分。[1] 同时，OKX 的 **Wallet API** 又把更底层的钱包能力拆成地址与账户创建、链上资产与币价查询、交易组装/签名/广播/追踪/历史，以及 token 信息、gas 价格、授权信息等辅助数据模块。[2]

所以，如果你问“**Agent Wallet 这个 API 有什么功能**”，最准确的回答不是一句“它能登录和交易”，而是：**它本质上是一组围绕钱包生命周期、账户地址、资产读取、交易执行、安全控制和执行结果追踪的 API 能力集合。**

## 二、Agent Wallet API 的标准功能边界

先按官方定义，把 Agent Wallet API 能力拆成最清楚的六个主域。

| 功能域 | 官方能力边界 | 对海豚社区的实际含义 |
|---|---|---|
| **1. 钱包创建与认证** | 邮箱登录即可创建钱包；身份验证后获取可用钱包上下文 [1] | 用户邮箱注册/验证码验证后，自动生成并绑定 Agent Wallet |
| **2. 账户与地址管理** | 地址与账户创建，多地址/子钱包管理，支持多链与多地址 [1] [2] | 获取 EVM / Solana 地址，后续用于资产、充值、转账、Swap |
| **3. 资产与账户状态读取** | 查询链上资产、余额、价格、组合视图 [2] | 钱包首页资产、聊天里“我的资产/余额/持仓”能力 |
| **4. 交易执行相关能力** | 交易组装、签名、广播、追踪、历史查询 [2] | Swap、转账、合约调用、交易回执、订单状态 |
| **5. 安全与风控能力** | 风险模拟、评分、黑名单拦截、风险代币提醒、异常阻断 [1] | 交易前风控、签名前风险提示、危险地址或危险 token 拦截 |
| **6. 常用链上辅助数据** | token 信息、gas 价格、授权信息等 [2] | 代币详情、授权检查、gas 展示、执行前准备数据 |

这六个功能域里，**1 和 2 是“钱包生成”的前提层**，**3 是“钱包可看”能力**，**4 是“钱包可动”能力**，**5 是“钱包可安全执行”能力**，**6 是“钱包执行所需的基础数据层”**。如果后面要做海豚社区自己的技能封装，这六层就是最底部的能力地基。

## 三、围绕“邮箱注册触发钱包生成”这条主链路，Agent Wallet API 实际上做了什么

如果完全贴着你的交互流程来理解，用户从邮箱注册进入之后，Agent Wallet API 至少需要承接下面这一条闭环。

| 流程阶段 | Agent Wallet API 实际职责 | 说明 |
|---|---|---|
| **邮箱输入** | 发起认证初始化 | 先发验证码，建立一次登录/注册会话 |
| **验证码校验** | 完成身份验证 | 验证成功后获取 access token 或钱包上下文 |
| **钱包生成/发现** | 创建账户或查询既有账户 | 如果是新用户就创建钱包，如果已有则拉已有账户 |
| **地址下发** | 返回链上地址 | 至少返回后续可用的 EVM / Solana 地址 |
| **会话建立** | 把钱包与 App 用户态绑定 | 让海豚社区后续所有能力都能基于这个钱包运行 |
| **钱包可读** | 查询余额、资产、价格、组合 | 支撑首页、聊天、钱包页的数据展示 |
| **钱包可写** | 发起转账、Swap、签名、广播 | 支撑海豚社区里的实际执行动作 |
| **结果回流** | 返回 txHash、订单状态、历史记录 | 支撑回执、任务状态和聊天续跑 |

你现在的方向是对的：**邮箱注册不是终点，而是 Agent Wallet 能力链的起点。** 也就是说，注册动作只是为了生成并激活一个“可读、可写、可风控、可回执”的执行账户。

## 四、结合 OKX 官方文档，Agent Wallet API 可以被理解成哪些具体功能

从官方文档的表达看，可以把 Agent Wallet API 进一步拆成“用户可感知功能”与“系统底层功能”两层来看。

### 4.1 用户可感知功能

| 用户看到的功能 | 背后的 Agent Wallet API 能力 |
|---|---|
| 邮箱注册后自动拥有钱包 | 认证初始化、验证、账户创建、地址生成 |
| 查看我的钱包地址 | 地址查询、账户详情读取 |
| 查看我的资产和余额 | 链上资产查询、币价查询、组合聚合 |
| 聊天里说“我的资产” | 钱包资产读取接口 + 上层意图路由 |
| 聊天里说“转 10 USDT 给某地址” | 转账构建、签名、广播、回执 |
| 聊天里说“把 100 USDT 换成 ETH” | 报价、交易构建、签名、广播、订单跟踪 |
| 查看这笔交易成功没 | 订单状态查询、交易追踪、交易历史 |
| 执行前提示风险 | 风险模拟、风险评分、地址/代币风险检查 |

### 4.2 系统底层功能

| 系统侧功能 | 作用 |
|---|---|
| **认证初始化** | 发送验证码，创建登录流上下文 |
| **认证验证** | 校验验证码，换取钱包上下文 |
| **账户创建** | 新用户第一次登录时生成钱包账户 |
| **账户列表** | 查询当前用户已有的钱包账户 |
| **地址列表** | 查询账户下多链地址 |
| **余额/资产查询** | 返回链上持仓与估值数据 |
| **价格查询** | 给资产估值或交易展示做支撑 |
| **交易组装** | 把转账/Swap/调用参数变成可签名交易 |
| **签名执行** | 在安全环境中完成签名 |
| **广播交易** | 把已签名交易发到链上 |
| **交易追踪** | 查询 txHash、订单状态、执行结果 |
| **交易历史** | 查询历史执行记录 |
| **Gas / Token / 授权信息** | 为交易准备阶段提供必要辅助数据 |
| **风险模拟与评分** | 在执行前决定是否放行、提示还是阻断 |

## 六、如果站在“海豚社区自我技能封装”的视角，Agent Wallet API 应该先被封成哪些基础技能

既然你接下来要围绕 Onchain OS 生态做“海豚社区自己的技能封装”，那么第一批不要贪多，我建议直接把 Agent Wallet API 封成下面五个基础技能模块。这样最稳，也最符合你现在的产品结构。

| 海豚社区自有技能名建议 | 底层依赖的 Agent Wallet API 功能 | 为什么要先封这个 |
|---|---|---|
| **钱包开通技能** | 认证初始化、认证验证、账户创建、地址下发 | 这是所有能力的起点，直接对应“邮箱注册触发钱包生成” |
| **钱包上下文技能** | 当前用户、钱包地址、账户状态、会话恢复 | 让聊天、钱包页、社区页都能共用统一钱包身份 |
| **钱包资产技能** | 余额、资产、价格、组合查询 | 支撑“我的资产”“我的持仓”“钱包资产页” |
| **钱包执行技能** | 转账、Swap、签名、广播、回执 | 支撑一句话交易、转账、链上动作闭环 |
| **钱包风控技能** | 风险模拟、风险评分、危险地址/危险 token 检查 | 保证执行动作不会变成无保护的裸调用 |

这五个技能里，**前两个是用户注册后立即生效的“基础技能”**，第三个是“钱包可读技能”，第四个是“钱包可写技能”，第五个是“钱包安全技能”。后面再往上，才是 DeFi 策略技能、聪明钱跟随技能、自动任务技能等更高层封装。

## 七、我建议你内部先采用的 Agent Wallet API 功能分组

为了后续开会、写方案、排研发任务都方便，我建议你们内部以后统一按下面这套口径叫，不要再把所有内容混成“钱包接口”。

| 内部统一叫法 | 包含的 Agent Wallet API 功能 |
|---|---|
| **A. 开户与认证 API** | 发验证码、校验验证码、创建账户、登录、登出、会话恢复 |
| **B. 钱包账户 API** | 账户列表、地址列表、多链地址、子钱包/多地址管理 |
| **C. 钱包读取 API** | 余额、资产、价格、持仓、组合价值、历史查询 |
| **D. 钱包执行 API** | 转账、Swap、合约调用、签名、广播、订单追踪 |
| **E. 钱包安全 API** | 风险模拟、风险评分、黑名单地址拦截、风险 token 告警、授权检查 |
| **F. 钱包辅助 API** | token 信息、gas 价格、授权信息、交易前辅助数据 |

只要这六类一分清楚，后面你做海豚社区自己的 Skill 层就很顺了。因为你会知道：**上层技能只是在编排这些基础 API，不是在重新发明钱包能力。**

## 八、现在最重要的判断

基于你当前的产品路径，我对“Agent Wallet API 有什么功能”给出的最核心判断是：

> **Agent Wallet API 的第一性功能不是交易，而是“给每个邮箱用户生成一个可认证、可持有资产、可发起链上动作、可被安全控制、可追踪执行结果的智能钱包账户”。**

在这个前提下，它的完整功能应该被理解成：

| 层级 | 核心问题 | Agent Wallet API 回答什么 |
|---|---|---|
| **身份层** | 这个用户怎么拥有钱包？ | 邮箱验证、账户创建、地址生成 |
| **状态层** | 这个钱包现在有什么？ | 资产、余额、价格、组合状态 |
| **动作层** | 这个钱包能做什么？ | 转账、Swap、合约调用、广播、历史 |
| **安全层** | 这个动作能不能安全执行？ | 风险模拟、评分、拦截、告警 |
| **结果层** | 执行以后怎么反馈？ | 回执、订单状态、历史记录 |
