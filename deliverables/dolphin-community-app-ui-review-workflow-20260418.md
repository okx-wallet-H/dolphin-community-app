# 海豚社区 App：UI 评审机制与交付节奏

**作者：Manus AI**  
**日期：2026-04-18**

## 一、评审机制

UI 设计师已经可以开工，但后续所有页面都必须纳入我主导的逐页评审机制。这里的关键不是让设计师自由出方案，而是要求其先基于既有页面识别问题，再在既定边界内提出修正建议。我会先做产品与实现双重复核，只有通过后，才允许进入下一页或进入代码实现。

## 二、单页交付格式

每一页都必须按统一格式提交，避免讨论发散。

| 提交项 | 要求 |
|---|---|
| 当前问题 | 明确指出高级感不足、统一性不足或信息层级混乱的位置 |
| 修正建议 | 只能在现有主路径和页面职责内提出 |
| 影响范围 | 说明仅影响样式、层级、留白、图标、文案轻重，不影响功能语义 |
| 风险点 | 说明是否可能触碰产品边界或开发结构 |

## 三、页面推进顺序

| 顺序 | 页面 | 我关注的通过标准 |
|---|---|---|
| 1 | 钱包页 | 资产总览与资产列表主次明确，次要模块被压低 |
| 2 | 币种明细页 | 账本页秩序感成立，状态与流向阅读顺序稳定 |
| 3 | 聊天页交易卡片 | 服务感强于工具台感，按钮主次清晰 |
| 4 | 赚币页 | 真实产品列表成为主角，页面不再像演示页 |

## 四、我的放行标准

| 维度 | 放行条件 |
|---|---|
| 产品正确性 | 不改变主入口、主链路和结果归属 |
| 视觉统一性 | 回到现有设计令牌体系，不另起新风格 |
| 落地可行性 | 可以直接在现有代码结构中实现 |
| 迭代纪律 | 当前页未通过，不进入下一页 |

## 五、交付节奏

本轮会采用“小步快跑、逐页冻结”的节奏。也就是说，钱包页通过后再看币种明细页，币种明细页通过后再看聊天页，最后才是赚币页。所有页面都通过后，才会进入 APK 前的整体验收阶段。

## 六、当前结论

UI 设计师现在可以从 **钱包页** 正式开工，但必须遵守我已经推送到仓库的开工说明和 GitHub 跟踪入口。我会继续担任唯一收口人，保证设计不会偏离产品方向。
