# H-Wallet 前端真实资产列表接入与联调结论

## 结论摘要

本次已将前端资产列表的**默认后端指向**收敛到当前已验证通过的 Agent Wallet 真实链路，并完成最小联调验证。当前可以确认两件事：第一，真实钱包地址 `0x65a92c1c5da328ae028e80c4fb2bfb223f652669` 对应的多链资产接口已经能够返回你刚转入的 **3 USDT**；第二，前端代码侧已补上当原生 App 未显式注入 `EXPO_PUBLIC_API_BASE_URL` 时的兜底逻辑，避免继续因为环境变量缺失或旧构建配置而落回错误域名或相对路径。

## 本次改动

| 文件 | 改动内容 | 目的 |
|---|---|---|
| `constants/oauth.ts` | 新增 `VERIFIED_AGENT_WALLET_API_BASE_URL = https://new-h-wallet-api.vercel.app`，并把 `getApiBaseUrl()` 的最终回退改为该已验证域名 | 确保原生 App 在未显式配置环境变量时，资产、登录、链上能力仍统一命中当前真实 Agent Wallet 后端 |
| `lib/_core/api.ts` | 将链名 `56` 的展示从 `BSC` 调整为 `BNB Chain` | 让用户在资产列表中更容易识别这笔 3U 的到账链别 |

## 联调验证结果

### 1. 类型检查

我已经对当前改动后的仓库执行了 `pnpm check`，结果为 **通过**，没有引入新的 TypeScript 编译错误。

我直接对当前已验证后端的 `/api/okx/mcp` 发起了资产查询，请求地址为：

> balance: `3`

这说明你刚转入的 **3U 已经能够被当前真实多链资产入口识别**，并且资产所在链别是 **BNB Chain**。

## 为什么这次改动能解决前端展示问题

之前的核心问题不是“链上没有到账”，而是“前端运行时不一定稳定命中已验证后端”。如果原生 App 构建时没有正确注入 `EXPO_PUBLIC_API_BASE_URL`，前端请求会退回到相对路径，这会导致移动端环境下的后端请求不稳定，进一步影响资产列表展示。现在把默认回退直接锁到已验证通过的真实后端后，钱包页请求真实资产数据的链路就统一了。
