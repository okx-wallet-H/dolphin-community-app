## 一、执行摘要

本次审查结论是：**仓库当前已经完成了 Agent Wallet 真实链路的核心修复，但生产环境仍存在“新旧域名并存、客户端链路未完全统一、前端业务回归与品牌一致性不足”的明显问题，尚不适合直接进入完整交付阶段。**

从最关键的 Agent Wallet 主链路看，新的服务端实现已经可以通过真实 OKX Agent Wallet 完成邮箱验证码发送与验证，并返回完整钱包结构，包括 EVM 地址与 Solana 地址。实测中，修复后的新生产别名 `new-h-wallet-api.vercel.app` 已成功返回如下真实结果：EVM 地址 `0x65a92c1c5da328ae028e80c4fb2bfb223f652669`、Solana 地址 `D7NLQ4Py73sCT56688m6TP9cjNMJjGqngqdrLsHUy55Y`，且响应字段明确包含 `mockMode: false` 与 `isNewWallet: true`。这说明**真实链路已经走通**。

但与此同时，用户原先使用的旧域名 `h-wallet-api-v2.vercel.app` 仍然表现出与新部署不一致的返回特征，说明当前线上环境并非单一可信入口，而是处于**域名与部署版本割裂**状态。对于涉及钱包创建、地址验证、后续测试 USDT 入金这类高风险操作，这种不一致是当前最大的交付阻塞点。

除链路问题外，仓库还存在一个明确的 TypeScript 编译错误，以及一批业务回归测试失败。更重要的是，若把项目目标定义为 **“海豚社区 App 中的 H Wallet 产品化交付”**，那么当前代码在品牌命名、登录后跳转逻辑、我的页数据真实性、行情页函数接入、自动任务与赚币页业务闭环等方面仍有较大整理空间。

## 三、关键审查发现

### 1. Agent Wallet 真实链路已经成功，但当前可信入口只有新生产别名

本次实测确认，修复后的服务端已经可以在真实 OKX Agent Wallet 流程下完成邮箱验证码校验，并返回完整双地址钱包结构。该结果来自新部署别名，而不是旧域名。

| 检查项 | 结果 |
|---|---|
| 发送验证码 | 成功 |
| 验证码校验 | 成功 |
| 返回 `mockMode` | `false` |
| 返回 `isNewWallet` | `true` |
| 返回 EVM 地址 | `0x65a92c1c5da328ae028e80c4fb2bfb223f652669` |
| 返回 Solana 地址 | `D7NLQ4Py73sCT56688m6TP9cjNMJjGqngqdrLsHUy55Y` |
| 当前可信域名 | `https://new-h-wallet-api.vercel.app` |
| 不应继续作为入金依据的旧域名 | `https://h-wallet-api-v2.vercel.app` |

这意味着此前“验证码校验返回单一旧地址结构”的主要问题，**并不是新代码本身未修复**，而更像是**旧域名仍指向旧部署或旧入口**。因此，后续凡是涉及测试 USDT 入金、地址确认、钱包验证，都必须以这次新部署返回的真实钱包地址为准，**不能再以旧域名返回结果为准**。

### 2. 当前最大的生产风险不是代码缺函数，而是“部署与域名不一致”

仓库根目录 `vercel.json` 当前只有最简 schema 配置，但本地 `vercel build --prod` 生成的 `.vercel/output/config.json` 中仍出现了：

```json
{ "handle": "filesystem" },
{ "src": "^/api(/.*)?$", "status": 404 },
{ "src": "/(.*)", "dest": "/" }
```

单看这份构建产物配置，确实会让人怀疑 API 被错误回退；但结合本次线上实测，更接近真实的结论是：**新生产别名上，专用 serverless 函数已生效；旧域名上，仍然不是同一套产物行为。**

也就是说，当前真正的问题已经从“函数代码是否已修复”转移为：

> **生产环境缺少单一可信域名与单一可信部署。**

只要客户端、测试人员、用户仍可能访问旧域名，就仍可能再次遇到旧结构响应、旧 cookie 行为、旧 CORS 头或旧逻辑，从而造成再次误判。

### 3. 仓库当前无法通过 TypeScript 类型检查，说明主干代码尚未处于可发布状态

运行 `pnpm check` 后，TypeScript 明确报错：

| 文件 | 位置 | 问题 |
|---|---|---|
| `server/_core/agent-wallet.ts` | 第 300 行 | `normalizeEmail(parsed.email)` 中 `parsed.email` 可能为 `undefined`，与函数签名不匹配 |

对应片段如下：

```ts
return {
  email: normalizeEmail(parsed.email),
  flowId: parsed.flowId.trim(),
  tempPubKey: parsed.tempPubKey.trim(),
  issuedAt: Number(parsed.issuedAt || Date.now()),
} as RequestState;
```

这类错误虽然修复成本很低，但性质上说明：**当前主干并未保持“随时可检查通过”的健康状态。** 在涉及钱包、登录、部署的项目里，这会直接降低发布可信度。

### 4. 自动化测试失败数量较多，说明“UI 已改完”并不等于“业务已对齐”

运行 `pnpm test` 后，共出现 **9 个失败测试**。这些失败并非全部都是坏事，其中一部分是测试断言仍停留在旧产品方向；但对项目管理来说，这同样说明当前代码与测试基线之间没有被及时同步，交付口径并不一致。

| 测试文件 | 失败点 | 暴露的问题 |
|---|---|---|
| `app-redesign.test.ts` | 登录后期待跳到 `/(tabs)/chat`，现代码跳到 `/(tabs)/wallet` | 业务目标变更后，测试与实现未统一 |
| `hwallet-ui.spec.ts` / `ui-polish-regression.spec.ts` | 聊天页断言中未找到 `getDexSwapQuote` | 当前聊天页实现已改为别的能力组合，测试未同步或业务链路变更未收口 |
| `hwallet-ui.spec.ts` | `app.config.ts` 未包含 `appName: "H Wallet"` | 品牌命名仍然不统一 |
| `ui-polish-regression.spec.ts` / `app-redesign.test.ts` | 我的页未命中 `H Wallet 用户` 文案 | 页面品牌与测试基线偏离 |
| `app-redesign.test.ts` | 自动任务页断言与当前页面结构不一致 | 自动任务业务与页面分布发生变化，但未做统一收口 |
| `lib/_core/api.test.ts` | 行情页未命中 `getPublicMarketSnapshot` | 行情能力函数名或调用链发生变化，测试仍基于旧实现假设 |

这类问题的本质不是“测试坏了”，而是：**产品目标、页面实现、自动化验收标准三者没有同步演化。**

### 5. 品牌命名仍然混用，影响产品一致性与验收标准

当前项目在品牌层面仍存在明显割裂。

| 文件 | 证据 | 问题 |
|---|---|---|
| `app.config.ts` | `appName: "海豚社区"`，`appSlug: "h3"` | 应用壳层仍是“海豚社区”命名 |
| `app/index.tsx` | 登录页标题与文案大量使用 `H Wallet` | 钱包主入口采用 H Wallet 品牌 |
| `app/(tabs)/profile.tsx` | 个人页用户名是 `海豚社区用户` | 个人中心仍偏社区侧命名 |

如果项目定位是“海豚社区 App 内的 H Wallet 模块”，那么这种混用并非绝对错误；但前提是必须在信息架构、页面标题、测试断言、App 壳配置中明确区分“母品牌”与“功能品牌”。目前仓库并未完成这件事，因此看起来更像是**历史状态残留**，而不是清晰设计后的双品牌架构。

### 6. 登录后的默认跳转已经改成钱包页，但测试和业务文档未同步

当前 `app/index.tsx` 中，登录成功后的跳转目标明确是：

```ts
router.replace("/(tabs)/wallet")
```

这和旧测试中期望的 `/(tabs)/chat` 不一致。从产品逻辑上说，跳到钱包页并非错误，尤其是在 Agent Wallet 刚创建完成、用户首先需要查看地址和资产时，这个设计更合理。但问题在于：**代码、测试、验收口径没有一起改。**

因此，这一项我判断为**“合理变更未完成全链路同步”**，而不是单纯代码 bug。

### 7. 我的页仍然主要是静态演示数据，真实性不足

`app/(tabs)/profile.tsx` 当前展示了大量固定文本与固定统计数据，例如：

| 字段 | 当前值 |
|---|---|
| 用户名 | `海豚社区用户` |
| 地址 | `0xA8F2...9C31` |
| 钱包总数 | `3 个` |
| 已启用提醒 | `12 条` |

这些数据并没有看到与真实登录态、真实钱包地址、真实提醒配置发生绑定。页面视觉上已经比较完整，但如果按“代码审查”的标准来看，这仍然属于**高保真静态页**，不是完整业务页。

对于验收来说，这类页面风险很大，因为它们很容易让人误以为“已经完成”。实际上，一旦进入真机验证或联调，很快就会暴露出数据不可追溯的问题。

### 8. 行情页与聊天页已经接入真实能力，但函数命名与回归基线脱节

从源码看，聊天页已经实际接入了一批真实能力，例如：

| 页面 | 已接入能力 |
|---|---|
| `app/(tabs)/chat.tsx` | `executeOnchainSwap`、`previewOnchainSwap`、`getAccountAssets`、`getMarketSnapshotByMcp`、`searchDeFiProductsByMcp`、`getSmartMoneyLeaderboardByMcp`、`getMemeScanListByMcp` |
| `app/(tabs)/market.tsx` | `getMarketSnapshotByMcp`、`getHotTokensByMcp` |
| `app/(tabs)/earn.tsx` | 真实 DeFi 搜索结果 + 跳转 AI 对话确认申购 |

也就是说，项目并不是“没有接业务”，而是**接入方式与测试预期使用的函数名、路径和页面归属发生了变化**。例如测试仍在找 `getDexSwapQuote`、`getPublicMarketSnapshot`，但现实现已经迁移到 MCP/OnchainOS 新接口组合。这类偏差如果不及时统一，会导致后续每次重构都在“代码对了、测试错了”与“测试对了、代码旧了”之间反复摇摆。

### 9. 代码中仍保留开发期 fallback 和宽松写法，生产安全性与可维护性需要收口

当前仓库仍存在多类应在交付前清理的实现模式。

| 类型 | 例子 | 风险 |
|---|---|---|
| 开发期密钥 fallback | `process.env.JWT_SECRET || 'h-wallet-dev-secret'` | 生产环境误配置时会退回弱默认值 |
| 大量 `any` / `as any` | MCP 服务、API 适配层中较多 | 会掩盖真实数据结构问题 |
| `mockMode` 兼容路径仍散落 | API 层、onchain 层仍有 mock 字段兼容 | 容易让前端误以为存在回退容错 |
| 控制台日志残留 | 例如 OAuth 初始化日志 | 生产日志会混入调试信息 |

这些问题单个看不致命，但它们共同说明：**当前仓库更接近“高强度改造中的工作分支”，而不是“交付前冻结分支”。**

| 记录钱包验证结果 | 将本次成功校验得到的真实钱包双地址写入联调记录，作为后续测试基准 |

| 项目 | 建议 |

## 五、建议的下一步执行顺序

我建议后续按以下顺序推进：

| 顺序 | 建议动作 | 目标 |
|---|---|---|
| 1 | 锁定唯一生产域名并同步客户端配置 | 先解决“请求到底打到哪”的根问题 |
| 2 | 修复 `pnpm check` 报错 | 恢复主干工程健康度 |
| 3 | 更新或重写失效测试断言 | 让测试重新成为可信验收工具 |
| 4 | 继续按 UI 审核工作流逐页推进 | 满足“完成一页—UI 审核—再领下一页”的规则 |
| 5 | 在钱包页后继续审查/对接下一页 | 保持视觉与真实业务双对齐 |
| 6 | 所有页面通过后再执行 APK 构建 | 避免带着已知缺陷进入打包阶段 |

## 六、最终判断

综合来看，**这个仓库并不是“方向错了”，而是“关键链路已经修通，但整个项目仍处于中后期收口阶段”**。最大的好消息是，最危险的部分——**真实 OKX Agent Wallet 创建与验证**——已经在新生产别名上得到实证成功；最大的坏消息是，**部署入口、品牌口径、测试基线和部分页面数据真实性仍未统一**。

因此，当前最合理的项目判断是：

> **可以继续推进，但必须先把“唯一可信线上入口”和“代码/测试/文案的一致性”收口，否则后续 UI 对接、测试入金和 APK 构建都会反复返工。**
