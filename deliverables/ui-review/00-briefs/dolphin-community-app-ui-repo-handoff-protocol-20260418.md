# 海豚社区 App：UI 仓库互传协作入口

**作者：Manus AI**  
**日期：2026-04-18**

## 一、目的

从本轮开始，海豚社区 App 的 UI 设计稿、负责人审核意见、以及前端接收说明，统一通过仓库传递，不再依赖项目公共文件区做最终审看依据。这样做的目的，是把每一轮设计提交都绑定到具体文件、具体提交记录和具体页面范围上，避免“文件可见但不可追踪”或“看过但无法复盘”的问题。[1]

这份文档既是设计师的提交入口，也是负责人审核和前端落地的流转规则。后续无论是 Naobanabn，还是其他设计协作对象，都必须遵守同一套仓库协作口径。[2] [3]

## 二、仓库内固定目录

仓库内的设计协作材料统一放在 `deliverables/ui-review/` 目录下，并按页面和轮次继续拆分。目录结构采用如下约定。

| 目录 | 用途 | 负责人 |
|---|---|---|
| `deliverables/ui-review/00-briefs/` | 冻结边界、开工说明、任务单、评审规则 | 总负责人维护 |
| `deliverables/ui-review/10-wallet/round-01/` | 钱包页设计稿、说明、修改记录 | UI 设计师提交，总负责人审核 |
| `deliverables/ui-review/20-token-detail/round-01/` | 币种明细页设计稿、说明、修改记录 | UI 设计师提交，总负责人审核 |
| `deliverables/ui-review/30-chat-card/round-01/` | 聊天交易卡片设计稿、说明、修改记录 | UI 设计师提交，总负责人审核 |
| `deliverables/ui-review/40-earn/round-01/` | 赚币页设计稿、说明、修改记录 | UI 设计师提交，总负责人审核 |
| `deliverables/ui-review/90-approved/` | 负责人签字通过后的终版审定稿 | 总负责人冻结，前端只认这里 |

这套分层的核心目的是把“在审稿”和“已通过稿”彻底分开。前端只能读取 `90-approved` 中的文件作为最终落地依据，不允许从过程中间稿直接实现。[1] [4]

## 三、单页提交格式

每一轮页面提交都必须包含同一组文件，文件不齐视为未提交。为避免命名混乱，建议使用下表格式。

| 文件类型 | 命名建议 | 要求 |
|---|---|---|
| 设计图 | `wallet-page-round01-v01.png` | 单页单图，必须是完整页面，不得只截局部 |
| 说明文档 | `wallet-page-round01-notes.md` | 说明本轮修改目标、设计判断和未解决问题 |
| 对照清单 | `wallet-page-round01-checklist.md` | 逐条对应任务单、边界文件和前轮反馈 |
| 审核结论 | `wallet-page-round01-review.md` | 仅由总负责人回写 |

如果是第二轮或第三轮，命名中的 `round01` 继续递增；如果同一轮内部又有修订版本，则在文件名尾部追加 `v02`、`v03`。这样每个页面的修改历史都能从仓库提交记录中直接回溯。[2] [3]

## 四、流转规则

仓库互传不是简单地“把图片扔进来”，而是必须按照页面冻结顺序推进。规则如下。

| 阶段 | 执行动作 | 输出物 | 是否可进入下一步 |
|---|---|---|---|
| 设计师开工 | 只处理当前获准页面 | 设计图 + 说明文档 + 对照清单 | 否 |
| 负责人初审 | 核对边界、语义、完成度和可落地性 | 审核结论文档 | 否 |
| 设计师修订 | 仅修负责人指出的问题 | 修订版设计图与说明 | 否 |
| 负责人复审 | 给出通过 / 不通过结论 | 终版审核结论 | 通过后才可进入下一页 |
| 前端接收 | 只读取 `90-approved` 审定稿 | 前端实现任务 | 仅对已通过页生效 |

这里最重要的纪律有两条。第一，**未通过当前页，不得领取下一页**；第二，**未进入 `90-approved` 的稿件，不得交给前端实现**。这两条是为了防止设计和开发再次并行发散，破坏统一风格与产品边界。[1] [4]

## 五、审核标准

总负责人审核时，统一按下表四个维度判断，不再因为风格喜好而放松产品纪律。

| 维度 | 通过标准 | 直接打回情形 |
|---|---|---|
| 产品边界 | 不新增入口、不改变主路径、不篡改结果归属 | 擅自加模块、加流程、改页面职责 |
| 语义可信度 | 币种、金额、状态、按钮语义一致 | BTC / ETH 混写、法币与币量单位混写 |
| 成品完整度 | 完整构图、文案自然、无半截截图感 | 展示稿不完整、文案机械、说明不全 |
| 前端可落地性 | 可映射到现有代码结构和设计令牌 | 脱离现有布局体系、引入新风格体系 |

在这四项里，**产品边界** 和 **语义可信度** 属于硬门槛；只要其中任意一项不满足，就直接判定为不通过。[1] [2]

## 六、当前执行顺序

当前仍按既定顺序推进，先钱包页，再币种明细页，再聊天交易卡片，最后是赚币页。[1] 也就是说，仓库互传机制建立后，**首个正式试运行页面仍然是钱包页 Round 1**，设计师不能跳过钱包页直接提交其他页面。

## 七、前端接收口径

前端接收设计稿时，不直接读取设计师原始说明，而是只读取以下三类文件：其一，`90-approved` 中的最终设计图；其二，对应页面的最终审核结论；其三，必要时由总负责人补写的落地说明。这意味着前端接收的是“已裁决后的设计”，而不是“待讨论的设计”，从而把实现风险压到最低。[3] [4]

## 八、我的批示

从现在起，海豚社区 App 的 UI 协作以仓库为唯一正式流转通道。项目公共文件区仍可用于临时沟通与备份，但**不再作为最终审核与前端接收依据**。所有设计师、审核人和前端同学，都必须以仓库文件、仓库目录和仓库提交记录为准。
