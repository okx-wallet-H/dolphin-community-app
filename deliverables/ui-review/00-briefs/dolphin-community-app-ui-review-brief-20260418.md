# 海豚社区 App：UI 审看约束与对接说明

**作者：Manus AI**  
**日期：2026-04-18**  
**对应提交：`c9a8c9e`**

## 一、对接原则

本轮不是让 UI 设计师重新定义产品，也不是开放式重做页面，而是基于已经推送到仓库的 MVP 实现进行 **视觉质感提升与页面收口**。产品结构、信息层级、执行边界与主链路已经由我先行确定，UI 设计师只能在既定边界内做增强，不能自行新增业务入口、改写交互主逻辑或把页面重新做成另一套产品。

换句话说，后续对接将由我主导，UI 设计师承担的是 **视觉审看、质感修正、局部层级优化**，而不是产品重构。

## 二、已冻结的产品边界

| 模块 | 已冻结结论 | 不允许 UI 自行改动的内容 |
|---|---|---|
| 钱包页 | 资产列表是主入口，点击币种进入对应账号明细 | 不允许改成弹窗、不允许把资产点击入口移除 |
| 币种明细页 | 成功与失败交易都沉淀到账号明细 | 不允许改回聊天页做主追单 |
| 聊天交易卡片 | 只承担意图承接、轻提示、执行入口 | 不允许重新做成复杂流程控制台 |
| 赚币页 | 只展示真实投资产品，无真实结果就显示空状态 | 不允许加入 demo 卡片、占位策略、假数据 |
| 设计风格 | 必须沿用现有统一设计令牌与页面体系 | 不允许另起一套配色、字号和圆角体系 |

## 三、允许 UI 调整的范围

UI 设计师可以调整的部分，应该集中在 **高级感、统一性、清晰度** 三个目标上，而不是功能发散。

| 页面 | 允许优化的点 | 优化目标 |
|---|---|---|
| 钱包页 | Hero 区质感、资产行留白、分组间距、箭头与点击态反馈 | 提升高级感与可点击识别度 |
| 币种明细页 | Hero 卡层次、记录卡状态视觉、信息密度、数值排版 | 强化“账号明细中心”定位 |
| 聊天页交易卡片 | 按钮主次、卡片层次、标签与说明文案的轻重 | 降低控制台感，增强服务感 |
| 赚币页 | 空状态品质、真实产品卡片的一致性、APR/TVL 信息层级 | 做到克制、可信、非演示化 |

## 四、我给 UI 设计师的明确限制

后续如果进入审看环节，我会要求 UI 设计师严格遵守下面的工作方式。

| 规则 | 说明 |
|---|---|
| 逐页审看 | 一次只审一个页面，不允许整包发散式返稿 |
| 先评后改 | 先指出现页面的问题，再提视觉修正建议，不能先随意出图 |
| 不改产品语义 | 不擅自改变“聊天页、钱包页、明细页、赚币页”的职责边界 |
| 不造新入口 | 不新增未确认的按钮、Tab、状态面板或浮层 |
| 统一令牌 | 所有优化必须回到现有 `manus-ui.ts` 体系里落地 |
| 最终复核 | UI 提案必须先经过我复核，确认不偏产品方向后才继续落代码 |

## 五、我会优先让对方看的页面顺序

| 顺序 | 页面 | 审看目标 |
|---|---|---|
| 1 | 钱包页 `app/(tabs)/wallet.tsx` | 把资产总览页做得更像正式金融产品首页 |
| 2 | 币种明细页 `app/token-detail/[symbol].tsx` | 把新页面收成高端账本页，而不是临时详情页 |
| 3 | 聊天页 `app/(tabs)/chat.tsx` | 把交易卡片压低“工具台感”，提高“服务感” |
| 4 | 赚币页 `app/(tabs)/earn.tsx` | 确保真实产品页干净、可信、不像演示页 |

## 六、给 UI 设计师的直接口径

> 当前仓库已经有可运行的 MVP 结构，你不需要重新定义产品。请在现有页面基础上，按统一风格做高端质感优化。不要改主路径，不要新增功能，不要加入演示型组件，不要让聊天页承担主追单逻辑。你的工作重点是把现有页面做得更成熟、更克制、更像正式产品。

## 七、我对下一轮工作的安排

下一轮如果继续推进，我会先基于当前提交产出页面截图和具体审看标注，再把其中一页交给 UI 设计师逐页审看，而不是把整套产品开放给对方自由发挥。这样可以保证方向始终由我收口，不会出现“设计稿看起来很漂亮，但产品结构被做偏了”的情况。
