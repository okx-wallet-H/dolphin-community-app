# Agent Wallet 能力清单与当前状态说明

本文基于当前 `dolphin-community-app-ci-build` 仓库内前端页面、服务端路由与链上执行实现进行整理，目标是把 **Agent Wallet 现在到底能做什么、哪些已经打通、哪些还只是半成品** 说清楚，便于你后续排优先级。

## 一、结论先看

当前这套 **Agent Wallet** 已经不是单一“登录钱包”能力，而是一个包含 **邮箱验证码登录、真实钱包地址下发、多链资产展示、链上 Swap 预执行/执行/回执、签名回调恢复、对话线程承接** 的组合能力。

但需要特别区分：**Swap 主链路已经具备较完整的服务端能力框架，而 Transfer（转账）链路目前更多停留在意图识别、签名确认承接和主线程恢复层，真实统一广播接口还没有完全补齐。** 因此，不能把“转账能力”与“完整可稳定执行的链上转账能力”画等号。

| 能力模块 | 当前状态 | 说明 |
|---|---|---|
| 邮箱验证码登录 | 已实现 | 前端只保留邮箱 + 验证码方式进入 Agent Wallet |
| 真实钱包地址返回 | 已实现 | 登录校验后返回 EVM 地址与 Solana 地址 |
| 演示钱包拦截 | 已实现 | 如果返回 `mockMode: true`，前端会阻止继续登录 |
| 会话恢复 | 已实现 | App 启动后会检查本地 session 并恢复登录状态 |
| 当前用户查询 | 已实现 | 已提供当前登录用户查询接口 |
| 登出 | 已实现 | 已提供退出登录接口 |
| 钱包资产总览 | 已实现 | 钱包页可展示多链地址与资产估值 |
| 多链真实资产查询 | 已实现 | 已接入 OKX Onchain / MCP 资产查询 |
| Swap 意图识别 | 已实现 | 聊天页可识别兑换请求并生成确认卡片 |
| Swap 预执行预览 | 已实现 | 可生成执行确认摘要 |

| Swap 回执查询 | 已实现 | 已提供 receipt / order 状态查询 |
| 签名回调桥 | 已实现 | 钱包确认后可通过回调返回 App |
| 对话线程恢复 | 已实现 | 签名完成、取消、失败都能回到聊天主线程 |
| 充值地址展示 | 已实现 | 聊天中可根据链路返回充值地址 |

## 三、已经具备的核心能力

### 1. Agent Wallet 登录与身份建立

当前登录首页已经被整理为 **单一路径的邮箱验证码登录**。用户输入邮箱后发送验证码，再提交验证码完成校验。校验成功后，前端会保存 session，并把返回的钱包信息写入本地存储。

这意味着，当前 Agent Wallet 的第一层能力不是传统用户名密码，而是：

| 登录能力 | 当前表现 |
|---|---|
| 发送验证码 | 已可调用标准路径发送邮箱验证码 |
| 校验验证码 | 已可调用标准路径完成校验 |
| 返回钱包地址 | 校验成功后下发 EVM 与 Solana 双地址 |
| 建立登录态 | 成功后写入 session token |
| 恢复登录态 | App 重启后可自动恢复 |

更关键的是，前端现在还增加了一层 **安全限制**：如果服务端返回的是 `mockMode: true`，应用会直接阻止登录，不允许把演示钱包当成真实钱包继续用。这一点非常重要，因为它把“登录成功”和“拿到真实钱包”强制区分开了。

### 2. Agent Wallet 钱包地址与账户上下文

当前 Agent Wallet 登录后，至少维护两条地址：

| 地址类型 | 当前状态 | 用途 |
|---|---|---|
| EVM 地址 | 已返回 | 用于 EVM 路径资产、Swap、EVM 地址识别 |
| Solana 地址 | 已返回 | 用于 Solana 路径资产与 Solana 相关能力 |

也就是说，这不是单链钱包，而是 **至少面向 EVM + Solana 的双地址账户上下文**。钱包页会把地址作为账户的一部分展示出来，聊天、资产、执行等能力也都会依赖这两个地址继续往下走。

### 3. 钱包资产与多链余额展示

钱包页当前已经具备 **总资产估值、多链地址资产分组、Token 余额、价格、估值** 等展示能力。当前实现并不是单纯本地模拟，而是优先通过 OKX Onchain / MCP 获取真实资产数据；只有在关键 OpenAPI 凭证缺失时，才会退回到 mock 或 demo 兜底。

这一部分可以理解为 Agent Wallet 的“看资产”能力已经具备以下层次：

| 资产能力 | 当前状态 | 说明 |
|---|---|---|
| 总资产估值 | 已实现 | 聚合各链资产后展示总价值 |
| 多链地址分组 | 已实现 | 按链与地址分组显示 |
| Token 余额 | 已实现 | 展示余额、价格与估值 |
| 风险 Token 标识基础 | 已实现 | 数据结构里已保留风险标识 |
| 实时更新 | 已实现 | 支持刷新与事件驱动更新 |
| 无地址空状态处理 | 已实现 | 未完成登录时直接进入空状态 |

### 4. Swap：预览、确认、执行、回执

这是当前 Agent Wallet 最像“可落地交易能力”的部分。

服务端的链上配置里，已经显式声明如下能力为开启状态：`preview`、`execute`、`receipt`、`assets`、`simulate`、`broadcast`。这说明后端已经把 **链上执行模型** 设计成一个完整能力集合，而不是单独一个接口。

从实际流程看，当前 Swap 至少包括以下阶段：

| Swap 阶段 | 当前状态 | 说明 |
|---|---|---|
| 意图识别 | 已实现 | 聊天中可识别兑换请求 |
| 预执行摘要 | 已实现 | 生成兑换确认卡片 |
| 参数风控校验 | 已实现 | 执行前会校验风险参数 |
| 执行幂等保护 | 已实现 | 重复执行会被幂等保护阻挡 |
| 执行任务记录 | 已实现 | 已有 tx record / log 体系 |
| 钱包确认 | 已实现 | 等待 Agent Wallet 确认 |
| 广播与执行阶段 | 已实现但仍在完善体验 | 代码已支持执行模型，但聊天续跑文案显示前端体验层仍在继续补齐 |
| 回执查询 | 已实现 | 可查 order / tx 状态 |

换句话说，**Swap 是当前 Agent Wallet 最核心、最接近闭环的一条能力线**。

### 5. 签名回调与 App 内恢复

当前 Agent Wallet 不是“外部签完就丢了”，而是已经做了 **签名回调桥**。也就是用户在钱包里确认、取消或者失败之后，应用可以通过回调拿到结果，再回到聊天线程继续承接。

这部分当前已经支持：

| 回调能力 | 当前状态 | 说明 |
|---|---|---|
| 回调参数解析 | 已实现 | 解析 ctx、status、signedTx、txHash、error |
| 上下文校验 | 已实现 | 防止回调结果串到别的待执行任务 |
| 成功恢复 | 已实现 | 确认成功后跳回聊天主线程 |
| 取消恢复 | 已实现 | 取消后也能回到主线程并重做分析 |
| 失败恢复 | 已实现 | 失败后能带错误信息回到主线程 |
| 本地待签名上下文管理 | 已实现 | 可保存、读取、清理待执行上下文 |

这一点很关键，因为它说明 Agent Wallet 当前不仅是“发起交易”，还具备 **交易确认后的应用内续跑能力**。

### 6. 聊天驱动的钱包执行体验

当前项目里，Agent Wallet 和聊天页是高度耦合的。聊天页不仅能识别资产、行情、充值、Swap、转账，还会把 Agent Wallet 作为执行容器来承接这些动作。

因此，Agent Wallet 当前的能力，不只是传统钱包 UI，而是更接近：

| 对话驱动能力 | 当前状态 | 说明 |
|---|---|---|
| 充值地址返回 | 已实现 | 可在聊天中返回充值地址 |
| 资产上下文回答 | 已实现 | 基于钱包上下文回答资产类问题 |
| Swap 确认卡片 | 已实现 | 在聊天中展示兑换确认卡片 |
| 转账确认卡片 | 已实现 | 在聊天中展示转账确认卡片 |
| 执行进度反馈 | 已实现 | 用多阶段状态提示当前执行进度 |

这意味着，**Agent Wallet 在产品形态上已经不是纯钱包，而是“对话式钱包执行器”。**

## 四、当前只是部分完成或仍有缺口的能力

### 1. 转账能力还没有完全闭环

虽然聊天页已经有 **转账意图识别、转账确认卡片、签名回调恢复**，但代码里已经明确写出：**“当前真实转账广播接口尚未接入”**。这说明目前的转账能力，更准确地说是“前半段流程已具备，后半段统一广播仍未完全打通”。

所以当前对转账能力应当这样理解：

| 转账环节 | 当前状态 |
|---|---|
| 地址与金额识别 | 已实现 |
| 待确认状态展示 | 已实现 |
| 钱包确认回传 | 已实现 |
| 主线程恢复 | 已实现 |
| 真实统一广播接口 | 未完整实现 |
| 完整回执持续追踪 | 仍需继续补强 |

### 2. 新旧部署域名仍可能带来混淆

前面我们已经验证过：**新链路返回的是真实 OKX Agent Wallet 地址，而旧域名并不一定指向同一条链路**。因此，从产品视角看，当前 Agent Wallet 的一个现实风险不是“能力不存在”，而是“同一 App 可能被旧域名或旧部署误导”。

这会直接影响：

| 风险点 | 影响 |
|---|---|
| 旧域名仍存活 | 可能继续返回旧结构或旧逻辑 |
| 客户端环境变量未统一 | 可能命中错误后端 |
| 用户复制旧地址测试 | 可能误把旧地址当真实钱包 |

### 3. 前端执行续跑体验仍有补强空间

虽然服务端 Swap 能力框架较完整，但聊天页里仍能看到一些“广播续跑能力正在接入中”的提示。这说明：

> **后端执行模型已经具备，前端在“确认后自动续跑、统一广播状态回显、最终回执体验”上仍有产品化补强空间。**

也就是说，现在更像是“核心能力已在，交互闭环还没有打磨完”。

## 五、如果只用一句话概括

当前 Agent Wallet 已经具备 **真实邮箱验证码登录 + 真实 EVM/Solana 地址下发 + 多链资产查看 + Swap 预览/执行/回执 + 签名回调恢复 + 对话式钱包交互** 这些核心能力；其中 **Swap 是最完整的主链路**，而 **转账广播、前端续跑体验、部署统一性** 仍是下一阶段最需要补强的地方。

## 六、建议你接下来重点盯的三件事

| 优先级 | 建议事项 | 原因 |
|---|---|---|
| P0 | 统一生产域名与客户端 API 基地址 | 先避免继续命中旧 Agent Wallet 链路 |
| P0 | 补齐真实转账广播与回执闭环 | 让 Transfer 不再只是“确认成功但未真正执行完成” |
| P1 | 打磨签名后自动续跑与结果展示 | 让 Swap / Transfer 从“可用”变成“顺滑可用” |
