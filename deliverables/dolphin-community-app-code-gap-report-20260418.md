# 海豚社区 App 仓库代码不足分析报告（2026-04-18）

作者：**Manus AI**  
日期：2026-04-18

## 一、审查结论

我已经结合当前仓库中的聊天主线程、Onchain 聚合入口、OKX 能力封装层、DEX 执行层以及既有交付文档，对 **海豚社区 App** 的 Agent Wallet 相关代码做了一轮针对性审查。结论可以概括为一句话：**项目已经具备 OKX OnchainOS / Skill 化能力层的雏形，但主链路仍未完全收敛，导致“看起来能跑”和“真实可验收”之间仍有明显差距。**

从架构角度看，当前仓库最大的问题不是“完全没有能力”，而是 **能力入口分散、旧逻辑残留过多、真实执行与 mock 回退边界不够清晰**。这会直接影响老板关注的三类核心演示：一句话交易、智能赚币闭环、自动任务联动。也就是说，现阶段最需要做的不是继续补零散功能，而是把 Agent Wallet 的意图理解、确认卡片、Skill 执行、回执追踪统一到一条稳定主路径上。

## 二、本轮重点审查范围

| 模块 | 文件 | 审查重点 |
|---|---|---|
| 聊天入口 | `app/(tabs)/chat.tsx` | 是否已经从本地零散分支切到统一意图路由与确认卡片模式 |
| 统一意图协议 | `lib/agent-wallet-intent.ts` | 是否支持固定关键词/模板触发，是否覆盖任意金额与常见交易表达 |
| 链上执行入口 | `api/onchain/index.ts` | 是否已经具备统一的预览、执行、回执边界 |
| 服务端交易实现 | `server/_core/dex-swap.ts` | 是否仍存在本地 parser、静默 mock、伪成功回退 |
| 能力桥接层 | `api/okx/mcp.ts`、`server/_core/okx-mcp-service.ts` | 是否具备统一 Skill 化别名层，是否真正成为主路径 |
| 既有交付材料 | `deliverables/*.md` | 是否已有明确验证结论、阻塞点与修正方向 |

## 三、已确认的积极进展

本轮并不是只发现问题，也确认了几项关键正向进展。首先，仓库中已经新增了 `lib/agent-wallet-intent.ts`，开始把 **swap / transfer / price_query / portfolio_query / earn_query / unknown** 统一成结构化意图协议，这说明项目已经开始朝“固定触发规则优先、未知意图再交给上层智能体”的正确方向前进。

其次，`app/(tabs)/chat.tsx` 已完成一次重要收敛：当前消息发送主流程已经优先调用 `detectAgentIntent()`，并基于 `buildConfirmCard()` 生成转账与兑换确认语义，再分别路由到资产、价格、赚币、Swap 等能力分支。并且我已完成本地 TypeScript 校验，`npx tsc --noEmit` 已通过。这说明 **新的统一意图协议至少在类型层面已经落地成功**。

再次，`api/onchain/index.ts` 已经提供了比较清晰的统一后端边界，包括 `preview`、`execute`、`receipt`、`assets` 等路由，同时还具备风险校验、幂等拦截、交易记录落库与日志追加能力。这部分设计方向是正确的，说明服务端并不是没有架构基础，而是 **前端主线程与底层能力层之间还没有彻底“单入口化”**。

## 四、当前仓库的核心不足

### 4.1 主入口虽已开始收敛，但仍保留大量旧分支与双轨路径

当前 `chat.tsx` 已经接入新的 `detectAgentIntent()`，这是本轮最重要的修复成果之一。但从整体实现看，聊天入口仍保留了旧的 `extractPriceSymbol()`、`isAssetIntent()`、`isDeFiIntent()`、`parseChatAiIntent()` 等多套 fallback 路径。也就是说，**现在不是单一主链路，而是“新协议 + 老分支并存”的双轨结构**。

这种结构短期内利于兼容，但长期会带来两个问题。第一，行为会越来越难预测，同一类请求到底走规则路由、老 heuristics 还是 AI fallback，会逐渐失去可解释性。第二，测试难以形成稳定验收口径，因为同一条自然语言输入在不同时间、不同环境下，可能落在不同分支上。

| 问题 | 当前表现 | 风险 |
|---|---|---|
| 主入口不唯一 | 新规则路由后面仍保留老 heuristics 与 AI fallback | 难以稳定复现问题与定义验收标准 |
| 行为可解释性不足 | 同一请求可能命中新规则，也可能滑入旧逻辑 | 不利于排查老板演示中的偶发失败 |
| 维护成本高 | 一类意图需在多个分支同步修补 | 后续 GLM 接入后更容易出现冲突 |

### 4.2 交易确认与真实执行边界仍然不够干净

虽然当前产品目标是“固定关键词/模板触发意图 → 生成确认卡片 → 用户确认后交给 OKX Skill 执行”，但从 `runSwapFlow()` 的实际代码看，当前前端在生成卡片前，就已经会调用 `previewOnchainSwap()`，并进一步以 best-effort 方式调用 `executeOnchainSwap()` 预构建执行请求。严格来说，这还不是完全干净的“先确认、后执行”边界，而更接近 **“先准备执行上下文，再通过卡片继续确认”**。

这并不一定是错误，但它意味着目前的“确认卡片”还没有完全退化成纯 UI 层，前端仍在承担一部分执行预处理职责。如果未来要满足你提出的“前端不要自己做交易引擎”，还需要继续把交易准备、签名请求构造和执行上下文管理更多地下沉到统一 Skill/Onchain 层。

### 4.3 服务端交易层仍然存在静默 mock 回退，验收可信度不足

`server/_core/dex-swap.ts` 是当前仓库最需要警惕的文件之一。这个文件里存在非常明确的 `mock` 设计：当真实 DEX 凭证不存在时，`getDexSwapQuote()` 会走 `buildMockQuote()`，`executeDexSwap()` 会走 `buildMockExecute()`，订单查询也会返回 mock 数据。更关键的是，这个回退并不是强制报错，而是默认继续返回“看起来像成功”的结构。

这会带来一个严重问题：**演示表面上闭环了，但实际上没有真实成交、真实广播与真实回执**。对于内部开发阶段，这种 fallback 有一定价值；但对于老板验收、融资演示或对外汇报，它会显著削弱可信度。

| 问题点 | 代码表现 | 直接影响 |
|---|---|---|
| 报价可 mock | 无真实凭证时直接 `buildMockQuote()` | 用户看到的价格不一定来自真实链路 |
| 执行可 mock | 执行失败或缺凭证时回退 `buildMockExecute()` | 容易误以为交易已完成 |
| 订单查询可 mock | `getDexSwapOrders()` 也能伪造成功态数据 | 回执链路的可信度被削弱 |
| 严格模式默认未强开 | 只有显式设置 `OKX_DEX_STRICT=1` 才会禁止静默回退 | 测试环境很容易“误验收” |

### 4.4 意图理解仍有两套系统并存，且服务端仍残留本地 parser

新建的 `lib/agent-wallet-intent.ts` 已经可以识别“换成”和“用 A 买 B”等表达，这本身是进步。但与此同时，`server/_core/dex-swap.ts` 里仍保留着另一套 `parseSwapIntent()`：它先走 OpenAI JSON parser，失败后再退到正则。换句话说，**目前仓库里至少同时存在“前端规则解析”和“服务端 AI/regex 解析”两套交易意图系统**。

这会形成典型的协议漂移风险。今天前端支持“用 100U 买 ETH”，并不代表服务端所有调用栈也一定按同样规则解析；明天若服务端正则或 AI schema 改动，又可能与前端协议重新失配。因此，从工程治理角度看，意图识别应最终只保留一套权威协议：前端只做触发与展示，服务端只认结构化 schema，不再自行发明另一套交易语言解析器。

### 4.5 币种与链路仍偏静态，离“任意金额、任意币种”还有距离

尽管新协议已经支持任意金额输入，但“任意币种”这件事在当前仓库里仍未真正成立。`chat.tsx` 里的兑换链路仍然依赖 `resolveSwapToken()`，而现有映射主要覆盖少数 EVM / Solana 常见币种。也就是说，**金额已开始泛化，但资产发现仍然偏静态**。

这会带来两类限制。其一，用户说的是合法币种，但由于本地 map 没有收录，就会被误判为“不支持”；其二，即使后端能力层已经能处理更多 token，前端也可能因为早期拦截导致请求根本到不了真正的 Skill 层。

### 4.6 链路分层较多，能力别名层尚未成为唯一事实来源

当前仓库已经有 `api/okx/mcp.ts` 与 `server/_core/okx-mcp-service.ts` 这样的统一能力桥接层，也有 `server/_core/onchain-os.ts` 这样的 Onchain 执行层，还存在 `server/_core/dex-swap.ts` 这种更底层的交易实现。问题不在于层次多，而在于 **这些层次之间的职责边界还没有完全清晰**。

目前的实际状态更像是多条链路并行存在：有的能力走 Onchain 统一入口，有的能力走 MCP alias，有的能力仍残留在本地 parser 与 mock fallback 中。这会导致同一类能力在多个位置重复维护，后续一旦修改费率、路由参数、错误模型或链映射，极易出现前后不一致。

### 4.7 118 与 120 两条演示链仍未形成真实业务闭环

根据既有测试进展文档，118 与 120 当前仍处于“看起来有界面与部分能力，但真实联动未打通”的状态。赚币方案已经可以拉出来，但“一键启动”还只是把草稿带回对话页；自动任务卡片已经有展示，但关键的策略写入与同步存储链路还没有真正贯通。这说明项目目前更接近 **原型演示**，而不是完整闭环。

| 清单项 | 当前状态 | 主要短板 |
|---|---|---|
| 117 一句话交易 | 已修复一部分触发能力，但尚未完成真实链路验收 | 仍受 mock、链路与资金条件限制 |
| 118 智能赚币 | 推荐能力可返回，但启动动作未真正落地 | 缺真实申购与状态持久化 |
| 119 资产总览 | 已基本可验收 | 属于当前最成熟的一条链路 |
| 120 自动任务 | 展示存在，但写入与同步链路不足 | 缺真正的任务持久化证据 |

## 五、本轮新发现与当前判断

本轮最大的正向变化，是 **117 的“不会识别老板句式”这一核心阻塞已开始被修复**。新的 `agent-wallet-intent.ts` 已把“用 100 USDT 买 ETH”这类表达纳入固定规则识别范围，聊天页也已经优先使用这一协议路由。因此，117 现在的第一层问题，已经从“识别不到”转变为“识别到了，但真实执行和验收链路仍未完全收口”。

这个变化很关键，因为它说明项目已经从“功能方向不对”进入“能力边界还需清理”的阶段。换句话说，**现在不是完全推倒重做，而是应该继续沿现有统一协议往下收敛**。

## 六、建议的修正优先级

我建议接下来的工作不要再平均用力，而是按下面的优先级推进。

| 优先级 | 建议动作 | 目标 |
|---|---|---|
| P0 | 把 `chat.tsx` 中旧 heuristics 逐步降级为兜底，不再与新协议并列 | 建立唯一主入口 |
| P0 | 明确“确认前只做预览，确认后才执行”的后端职责边界 | 消除前端交易引擎感 |
| P0 | 在测试/演示环境强制开启严格模式，禁止静默 mock 成功 | 提高验收可信度 |
| P0 | 统一服务端交易 schema，停止多套 parser 并存 | 防止协议漂移 |
| P1 | 以 token discovery / metadata 取代前端静态币表主路径 | 向任意币种能力演进 |
| P1 | 打通赚币启动 → 自动任务写入 → 状态同步 | 修复 118/120 的业务断点 |
| P1 | 增加端到端验证清单与可复现测试脚本 | 降低回归风险 |
| P2 | 接入 GLM 5.1 作为 unknown intent fallback | 让规则与智能体形成清晰分层 |

## 七、当前总体评价

如果以“代码有没有继续推进”来评价，这个仓库是有进展的；如果以“是否已经达到老板要看的稳定演示级别”来评价，目前还没有完全达标。更准确地说，项目目前处在 **架构方向已纠偏、核心能力正在收敛、但真实闭环尚未完全形成** 的阶段。

因此，我的总体判断是：

> **这不是一个需要推翻重做的仓库，而是一个需要迅速完成主链路收敛、清理 mock 与双轨逻辑、补齐关键闭环证据的仓库。**

只要接下来继续围绕统一意图协议、统一确认卡片、统一 Skill 执行边界推进，117/118/120 这三项仍然有机会在较短周期内显著改善；但如果继续保留多套入口、多套 parser、多层静默回退并行存在，后续每修一个点都可能在别处重新失效。

## 八、建议立即跟进的三件事

| 顺序 | 动作 | 原因 |
|---|---|---|
| 1 | 继续把 Swap 真正收敛到“确认后执行”的单链路 | 这是老板最直观看到的能力 |
| 2 | 强化非真实链路的错误暴露，不再伪装成功 | 避免演示阶段误判 |
| 3 | 先补 118/120 的持久化与联动证据，再统一回归测试 | 让演示从“有页面”变成“有闭环” |
