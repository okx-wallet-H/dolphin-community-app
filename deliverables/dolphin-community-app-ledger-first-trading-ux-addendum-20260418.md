# 海豚社区 App：以账号明细为主的交易体验收口建议

**作者：Manus AI**  
**日期：2026-04-18**

## 一、最新产品口径

根据你刚刚连续确认的产品边界，海豚社区 App 的交易体验应进一步收口为以下模式：**用户触发交易后，不依赖聊天主线程做二次确认或持续追单；交易由 Agent 执行；最终结果以账号明细为主展示；若结果未正常到达，再由人工客服兜底处理。**

在这个口径下，**失败交易也必须进入账号明细**，而不是只在聊天里提示失败。因为从产品稳定性与客服协同角度看，用户最需要的是一个统一、可回看的结果落点，而不是依赖聊天线程是否恰好收到了最终回执。[1] [2] [3]

## 二、当前代码现状与新口径的偏差

当前仓库的偏差主要体现在两个层面。首先，`app/(tabs)/chat.tsx` 仍然承担了过多交易生命周期责任：它不仅负责触发交易，还直接串联预览、执行、签名回调恢复以及回执轮询。这说明聊天页目前仍被当成交易执行主线程，而不是单纯的触发入口与轻提示入口。

其次，`app/(tabs)/wallet.tsx` 当前仍然只是**资产总览页**。页面已经有总资产估值、多链地址、授权摘要、自动任务与资产列表，但并没有真正的**账号明细 / 交易明细**区块。因此，如果按照你最新的产品口径推进，当前最需要补的不是聊天页更多提示，而是钱包页或其下级页面的**交易明细承接能力**。

| 模块 | 当前现状 | 与新口径的偏差 | 收口方向 |
|---|---|---|---|
| 聊天页 `app/(tabs)/chat.tsx` | 负责触发、执行、续跑、查回执 | 聊天页越权承担交易生命周期 | 缩减为“发起入口 + 轻提示”，不再做主追单界面 |
| 钱包页 `app/(tabs)/wallet.tsx` | 仅展示资产、授权摘要、任务卡片 | 缺少账号明细承接层 | 新增交易明细列表或独立明细页 |
| 签名桥接与回调层 | 围绕 `signedTx / txHash / orderId` 恢复主线程 | 仍是旧的兼容执行心智 | 明确降级为兼容层，不作为默认产品路径 |

## 三、建议采用的最小账号明细模型

如果你要的是能快速落地、又不让系统过度复杂的版本，我建议账号明细先只保留一套**最小状态模型**。这个模型的目标不是完整复刻链上浏览器，而是让用户能在统一入口看到“这笔交易有没有成、失败了没有、失败原因是什么”。

| 字段 | 是否必需 | 说明 |
|---|---|---|
| 记录类型 | 必需 | 闪兑、转账、赚币申购、赎回等 |
| 发起时间 | 必需 | 用户触发执行的时间 |
| 支出资产 | 必需 | 例如 `100 USDT` |
| 获得资产 | 推荐 | 例如 `ETH`；失败时可保留目标资产 |
| 状态 | 必需 | `处理中` / `成功` / `失败` |
| 结果摘要 | 必需 | 成功摘要或失败原因简述 |
| 详情标识 | 推荐 | `txHash` 或 `orderId`，仅用于详情展开 |

在这套模型里，**失败记录必须和成功记录同权进入账号明细**。这样做有三个直接好处。第一，用户不需要回忆自己是在聊天里、社区里还是其他入口发起的交易；第二，客服处理问题时有统一对账入口；第三，即便聊天回执丢失，账号明细仍能成为主事实来源。

## 四、聊天主线程应删减到什么程度

按你现在的产品判断，聊天页并不需要承担“确认收到结果”的职责。更准确地说，聊天主线程只需要做三件事：接住用户意图、提示“已发起执行”、引导用户去账号明细查看结果。至于结果是否成功、失败、延迟或未达，都不必在聊天里形成复杂追单链路。

因此，当前应删除或弱化的逻辑主要是以下三类。

| 需删减的逻辑 | 原因 | 建议处理 |
|---|---|---|
| 聊天里二次确认交易结果 | 与“无需聊天确认收到”冲突 | 删除结果确认交互 |
| 聊天主线程持续轮询并追单 | 与“结果以账号明细为主”冲突 | 轮询结果写入明细存储，不强依赖聊天展示 |
| 聊天里作为失败主承接界面 | 与“失败也展示账号明细”冲突 | 聊天仅提示“可前往账号明细查看结果” |

这并不意味着聊天完全不显示任何内容，而是意味着聊天只保留**轻量状态提示**，例如“交易已发起，请到账号明细查看结果”。一旦这么收口，用户心智会明显更稳定：聊天负责发起，账号明细负责记账，人工客服负责异常兜底。

## 五、对当前仓库的直接整改建议

从代码层面看，最应该优先推进的不是继续补强聊天页的执行状态机，而是新增一个真正的**交易明细承接层**。这可以是钱包页中的新分区，也可以是从钱包页进入的“账号明细”二级页面。无论采用哪种形式，关键是把交易成功与失败都沉淀成同一类记录。

我建议按如下顺序推进。

| 优先级 | 建议 | 目的 |
|---|---|---|
| P0 | 在钱包页新增“账号明细 / 交易记录”入口或列表 | 为成功与失败结果提供统一落点 |
| P0 | 定义统一交易记录模型，至少包含类型、时间、状态、摘要 | 让前端、服务端、客服用同一语言对账 |
| P0 | 将聊天页的追单逻辑降级为轻提示 | 避免聊天继续充当交易结果主界面 |
| P1 | 将签名回调与续跑逻辑标注为兼容层 | 防止旧链路继续被误认为主路径 |
| P1 | 为失败记录预留客服处理字段或工单链接位 | 便于异常场景快速转人工 |

## 六、结论

所以，针对你刚刚这句“**如果交易失败，就展示账号明细**”，我的判断是：**这是对的，而且应该升级为统一原则——无论成功还是失败，都展示账号明细；聊天不负责最终收口，人工客服负责异常兜底。**

这会让海豚社区 App 的产品职责更清晰：

> **聊天负责触发，Agent 负责执行，账号明细负责落账，客服负责兜底。**

从当前仓库现状看，真正要补的也正是这一步：不是再做一层聊天确认，而是尽快把“账号明细”做成交易结果的主承接界面。
