> 版本：V2.0  
> 适用范围：H Wallet Expo / React Native App（iOS、Android、Expo Web 同步参考）  
> 文档属性：产品级设计规范，直接替换旧版设计说明  
> 更新时间：2026-04-15  
> 作者：**Manus AI**

## 一、设计目标与修订背景

H Wallet 当前已经从单纯的链上工具入口，升级为围绕**AI 对话、链上钱包、智能赚币、社区发现与个人中心**构成的完整移动端产品。因此，设计规范不能再停留在“页面还原说明”层面，而必须升级为一套可被产品、设计、前端与后端共同执行的**设计系统文档**。新版规范将统一品牌调性、视觉语言、组件定义、页面布局、交互反馈与亮暗色策略，确保后续功能扩展不会破坏整体一致性。

本次文档修订有两个直接触发因素。其一，仓库中原有 `design.md` 已经初步确立了**浅色、白底、柔和紫色渐变、玻璃卡片、中文金融产品风格**的方向，但对于设计 token、组件状态、动效规则、跨页面布局约束仍然不够完整。其二，老板已明确提出 4 项页面级改动要求，必须立即纳入新的标准规范，而不是仅作为临时改动存在。

| 修订来源 | 对设计系统的影响 | 本文处理方式 |
|---|---|---|
| 现有 `design.md` | 已建立浅色紫系、玻璃卡片、中文层级等基础 | 作为新版视觉基线保留并系统化 |
| 老板最新 4 项要求 | 涉及登录、首页导航、钱包头部、社区顶部搜索 | 直接写入页面布局规范与组件规范 |
| 移动钱包最佳实践 | 强调移动端优先、原生感、安全感、简洁性 | 吸收为整体风格与交互原则 [1] [2] |
| React Native 设计系统经验 | 要求 token、主题、组件、交互成体系定义 | 纳入设计 token 与组件库章节 [3] [4] |

## 二、外部参考与设计结论

H Wallet 的目标不是复制任何单一钱包产品，而是在“**轻盈可信的 AI 钱包**”这一方向上，吸收成熟移动钱包和 React Native 设计系统的优点。Rainbow Wallet 在公开仓库中将自己定义为“the Ethereum wallet that lives in your pocket”，其核心含义是**移动端优先、单手使用、品牌识别集中、避免桌面式复杂层级**[1]。Uniswap Wallet 则直接提出“simplest, safest, and most powerful self-custodial crypto wallet”，强调移动钱包在设计上必须同时满足**简洁、可信、安全、能力感**[2]。

与此同时，Tamagui 的设计系统指南强调，设计系统应从 **tokens、themes、fonts 与 components** 的统一结构出发，而不是散落在页面中以局部样式常量维护[3]。React Native Paper 则说明一个高质量的移动端组件体系必须覆盖**原生感、交互细节、主题切换与可访问性**，否则 UI 只会“看起来像 App”，却无法“用起来像 App”[4]。基于这些参考，新版 H Wallet 设计系统将坚持以下判断：

> **H Wallet 不是深色 Web3 控制台，也不是信息密度过高的交易终端，而是一款面向主流用户的“对话式、可信赖、可执行”的移动端 AI 钱包。**

| 参考项目 | 可借鉴结论 | 对 H Wallet 的落地方向 |
|---|---|---|
| Rainbow Wallet [1] | 强移动端属性、品牌识别集中、减少分散入口 | 默认首页聚焦 AI 对话，不堆叠杂乱头部入口 |
| Uniswap Wallet [2] | 简洁、安全、强能力感并存 | 页面结构简洁，但关键操作反馈要完整 |
| Tamagui [3] | 设计系统必须 token 化、组件化 | 统一颜色、字体、圆角、间距、阴影语义 |
| React Native Paper [4] | 原生感、主题、交互、可访问性都要被规范 | 补全输入、按钮、列表、反馈、暗色模式规则 |

## 三、品牌调性与整体风格定义

H Wallet 的品牌调性定义为：**可信、轻盈、智能、克制、未来感适中**。这里的“可信”意味着页面必须像金融产品一样稳定、明确、层级清晰；“轻盈”意味着界面要留白充足、色彩柔和、组件边界不沉重；“智能”意味着 AI 相关模块必须自然成为产品核心能力，而不是附属入口；“克制”意味着避免夸张霓虹、赛博朋克、过强发光特效；“未来感适中”则要求保留科技感，但始终让信息可读性和操作安全感优先。

在视觉方向上，H Wallet 应采用**暖白底 + 浅紫渐变 + 白色玻璃卡片 + 低饱和功能色点缀**的统一体系。背景层不是纯白，而是带有极轻的紫色过渡和局部柔光；卡片不是完全实体块，而是轻透明、弱描边、柔阴影的“轻玻璃”结构。品牌色必须集中使用在主按钮、激活态、关键数字高亮、图标托盘和少量重点模块上，而不能全屏大面积使用纯紫高饱和色。

| 维度 | 设计定义 | 禁止事项 |
|---|---|---|
| 品牌气质 | 智能金融、移动优先、AI 驱动、可信轻盈 | 深色霓虹、重金属、黑客风 |
| 视觉重心 | 信息清晰、层级明确、留白充分 | 元素过多、头部拥挤、密集栅格 |
| AI 感知 | AI 是默认入口与核心能力 | AI 只做角落功能按钮 |
| 钱包感知 | 资产、收益、地址、风险均要明确表达 | 只做概念展示、不提供状态与反馈 |
| 社区感知 | 发现与搜索优先，内容和信号辅助 | 复杂论坛式堆叠、纯内容流无目标 |

## 四、信息架构与导航原则

H Wallet 的默认首页定义为**对话页**，而不是钱包页。原因很明确：该产品的第一心智不是“资产列表”，而是“用户可以直接对 AI 说需求，并获得价格查询、转账、赚币、追踪与操作建议”。因此，信息架构必须围绕“对话即入口，钱包即底座，赚币与社区即能力扩展”展开。

底部主导航维持五栏结构：**钱包、行情、对话、赚币、我的**。其中“对话”是默认首页与产品主入口；“社区”不作为底部常驻 Tab，而是从对话场景或推荐链路中分流进入的功能页。这样可以减少主导航复杂度，让用户始终能在底部完成核心任务切换，同时避免“社区”和“赚币”抢占 AI 首页认知。

| 导航层级 | 页面 | 角色定义 | 入口原则 |
|---|---|---|---|
| 一级主 Tab | 钱包 | 资产总览与钱包操作底座 | 固定出现在底部导航 |
| 一级主 Tab | 行情 | 价格、涨跌、市场浏览 | 固定出现在底部导航 |
| 一级主 Tab | 对话 | 默认首页，AI 操作入口 | 启动后优先进入 |
| 一级主 Tab | 赚币 | AI 智能赚币产品线 | 固定出现在底部导航 |
| 一级主 Tab | 我的 | 账户、偏好、设置与帮助 | 固定出现在底部导航 |
| 二级功能页 | 社区 | 搜索发现、地址/代币/合约检索、策略观察 | 不占用主 Tab，按任务进入 |

## 五、老板最新 4 项修改要求的正式落版

老板提出的 4 项修改，不应被视为页面微调，而应成为 H Wallet V2 的正式设计规范。原因在于，这 4 项改动本质上都在强化一个共同目标：**减少冗余入口、强化默认任务、优化页面主语义**。

### 5.1 登录页改动

登录页必须取消密码输入框，统一为**邮箱 + 验证码登录 / 注册**。这意味着 H Wallet 的登录心智不再是传统账户密码体系，而是**轻注册、轻登录、低阻力进入钱包**。页面文案、输入区布局、交互状态、校验提醒都要围绕双字段模型重新设计。

### 5.2 对话页头部改动

对话页作为默认首页，必须移除顶部的“钱包入口”和“我的”两个按钮，仅保留**页面标题 / 品牌标识 + 对话/社区切换（如保留）或更轻量的导航表达**。核心目标是让用户第一眼进入页面时，焦点落在快捷能力区、消息流与输入区上，而不是被三段式头部干扰。

### 5.3 钱包页头部改动

钱包页右上角不再使用铃铛图标，而是明确改为**返回按钮**，点击后返回对话页。钱包页因此被重新定义为“从对话任务进入的钱包底座页”，而非一个与首页完全并列、互不关联的孤立空间。头部语义必须清晰体现“返回 AI 首页”的路径感。

### 5.4 社区页头部改动

社区页顶部原有“社区 / Agent 中枢”标题区域改为**搜索框**，搜索对象包括**代币、合约、地址**。这会直接改变社区页的定位：从偏“任务控制台”转向“搜索驱动的发现页”。搜索框必须成为页面首要控件，并承担全局发现入口角色。

| 修改项 | 旧状态 | 新规范 | 设计目标 |
|---|---|---|---|
| 登录页 | 邮箱 + 验证码 + 密码 | 仅保留邮箱 + 验证码 | 降低登录阻力 |
| 对话页头部 | 左钱包入口 / 中切换 / 右我的 | 去掉左右按钮，头部减负 | 强化默认首页聚焦 |
| 钱包页右上角 | 铃铛或消息类图标 | 明确返回对话页 | 建立主任务回流路径 |
| 社区页顶部 | 标题型头部 | 搜索框型头部 | 强化发现与检索 |

## 六、配色系统

H Wallet 的色彩体系采用“**品牌紫 + 暖白底 + 低饱和功能色**”的方式构建。配色不追求交易所式高对比刺激感，而追求“安全、亲和、科技感适中”的金融产品体验。颜色必须优先以**语义 token**定义，而不是零散十六进制值散落在页面中。

H Wallet 允许使用渐变，但渐变必须克制且有明确角色。主按钮、Hero 卡片、重要图标托盘可使用渐变；正文卡片、搜索框、列表行、输入区域不得大面积使用强渐变，以避免影响可读性。

## 七、字体规范

H Wallet 的排版风格应接近 iOS 原生金融类应用，强调**高可读性、柔和层级、数值稳定感**。中文字体以系统字体为基础，避免为追求个性而牺牲清晰度。页面中的数字、金额、APR、涨跌幅等金融信息必须具备更稳定的字重与间距感。

当前代码已存在跨平台字体策略：iOS 使用 `system-ui`、Web 使用系统字体栈，这一方向应继续保留，并在规范中进一步明确字号、行高、字重和用途[3] [4]。

| 层级 | Token 建议 | 字号 | 行高 | 字重 | 用途 |
|---|---|---:|---:|---:|---|
| 展示大标题 | `font.display` | 28 | 34 | 700 | 登录页品牌、核心主标题 |
| 页面标题 | `font.h1` | 24 | 30 | 700 | 页面主标题、Hero 标题 |
| 模块标题 | `font.h2` | 20 | 26 | 700 | 卡片标题、模块标题 |
| 次级标题 | `font.h3` | 17 | 24 | 600 | 列表标题、表单标题 |
| 正文主文 | `font.body` | 15 | 22 | 400 | 大多数说明文案 |
| 正文强调 | `font.bodyStrong` | 15 | 22 | 600 | 重点说明、按钮文案 |
| 辅助文案 | `font.caption` | 13 | 18 | 400 | 标签、副信息、状态说明 |
| 最小注释 | `font.mini` | 12 | 16 | 400 | 占位、时间、次级标签 |
| 数字强调 | `font.numeric` | 依场景 17-32 | 1.15 倍字号 | 700 | 金额、收益、涨跌 |

### 7.1 字体族建议

| 平台 | 字体策略 | 说明 |
|---|---|---|
| iOS | `system-ui` / SF Pro 系 | 保持原生金融 App 观感 |
| Android | 系统无衬线字体 | 保证兼容性与清晰度 |
| Web | `system-ui, -apple-system, BlinkMacSystemFont, Segoe UI, Roboto, Helvetica, Arial, sans-serif` | 与当前代码基线保持一致 |

## 八、间距体系与圆角阴影

间距系统必须采用统一基数，避免页面上出现“同级模块间距不一致、卡片内边距忽大忽小、输入框高度各不相同”的问题。推荐以 **4pt** 为基础单位，构建 4、8、12、16、20、24、32 的递进体系。整体节奏应服务于“轻盈、可呼吸”的移动端体验，而不是追求信息极致压缩。

| 语义角色 | Token 建议 | 数值 | 用途 |
|---|---|---:|---|
| 最小间距 | `space.1` | 4 | 图标与字微调 |
| 超小间距 | `space.2` | 8 | 标签内边距、细小分组 |
| 小间距 | `space.3` | 12 | 列表行内小间距 |
| 标准间距 | `space.4` | 16 | 页面常规内边距、卡片内容间距 |
| 中间距 | `space.5` | 20 | 模块上下分隔 |
| 大间距 | `space.6` | 24 | 页面主要分区间隔 |
| 超大间距 | `space.7` | 32 | Hero 与主模块间距 |

### 8.1 页面边距规范

| 场景 | 左右边距 | 顶部边距 | 底部安全区 |
|---|---:|---:|---:|
| 普通页面 | 16 | 12-16 | 跟随设备安全区 |
| Hero 页 | 16 | 8-12 | 底部导航之上需额外预留 92-108 |
| 表单页 | 20 | 16-24 | 键盘弹起后保证主按钮可见 |
| 对话页 | 16 | 8-12 | 输入区与底部导航分层，避免重叠 |

### 8.2 圆角与阴影规范

| 角色 | Token 建议 | 数值 / 风格 | 用途 |
|---|---|---|---|
| 小圆角 | `radius.sm` | 12 | 标签、小按钮、小卡片 |
| 中圆角 | `radius.md` | 16 | 输入框、列表卡、普通按钮 |
| 大圆角 | `radius.lg` | 20 | 模块卡片、分区容器 |
| 超大圆角 | `radius.xl` | 24 | Hero 卡、主表单卡 |
| 胶囊圆角 | `radius.pill` | 999 | Segmented Control、胶囊按钮 |
| 轻阴影 | `shadow.soft` | `0 8 24 rgba(124,58,237,0.08)` | 普通卡片 |
| 强调阴影 | `shadow.emphasis` | `0 12 32 rgba(124,58,237,0.14)` | Hero 卡、浮层按钮 |
| 聚焦阴影 | `shadow.focus` | `0 0 0 4 rgba(124,58,237,0.10)` | 输入聚焦、可编辑状态 |

## 九、组件库规范

H Wallet 的组件必须从“**语义统一**”角度出发。也就是说，按钮不只是一个有颜色的矩形，输入框不只是一个边框容器，卡片也不只是一个背景块。每个组件都必须有明确的角色、尺寸、层级、状态和在不同页面中的使用边界。

### 9.1 按钮

按钮分为主按钮、次按钮、描边按钮、文本按钮与图标按钮五类。主按钮只用于单个区域中的首要操作，例如“继续进入”“发送”“确认执行”“立即申购”。描边按钮用于次操作，例如“查看详情”“稍后处理”。文本按钮用于辅助链路，如“重新发送验证码”。图标按钮必须仅出现在功能明确且用户能够预期其行为的位置。

| 按钮类型 | 视觉样式 | 高度 | 使用场景 | 状态 |
|---|---|---:|---|---|
| 主按钮 | 品牌紫渐变 + 白字 + 柔光阴影 | 48-52 | 登录提交、发送消息、确认执行 | 默认 / 按下 / 加载 / 禁用 |
| 次按钮 | 白底 + 深色字 + 轻描边 | 44-48 | 次级跳转、查看详情 | 默认 / 按下 / 禁用 |
| 描边按钮 | 白底 + 紫边 + 紫字 | 40-44 | AI 卡片辅助操作 | 默认 / 按下 |
| 文本按钮 | 无底色 + 紫字 | 自适应 | 链接、重发验证码 | 默认 / 按下 / 禁用 |
| 图标按钮 | 圆形或方圆角浅底 | 36-44 | 返回、搜索、刷新、关闭 | 默认 / 按下 |

### 9.2 输入框

输入框统一采用**白底、大圆角、轻描边、聚焦时紫色光晕**的样式。验证码输入框、搜索框、聊天输入框、邮箱输入框虽然尺寸与布局不同，但必须共享同一套语义 token。错误态不得仅用文字提示，边框与说明文案都应同步变色。

| 输入类型 | 结构 | 高度 | 规则 |
|---|---|---:|---|
| 邮箱输入框 | 单行输入 + 左侧图标可选 | 52 | 使用白底与轻边框 |
| 验证码输入框 | 单行输入 + 右侧发送按钮 | 52 | 按钮与输入区一体化布局 |
| 聊天输入框 | 宽底板 + 右侧圆形发送按钮 | 52-56 | 固定底部，支持多状态 |
| 搜索框 | 搜索图标 + 占位 + 右侧快捷清除 / 扫码可扩展 | 44-48 | 社区页顶部主控件 |

### 9.3 卡片

卡片是 H Wallet 最核心的容器组件。卡片需分为 Hero 卡、信息卡、操作卡、列表卡、消息卡五类。所有卡片默认使用白色或半透明白色背景，并用轻描边与柔阴影保持层次，而不是依赖深色块和强投影。

| 卡片类型 | 样式特征 | 使用场景 |
|---|---|---|
| Hero 卡 | 渐变背景、较大圆角、重点数字 | 钱包总资产、核心赚币推荐 |
| 信息卡 | 白色半透明、留白较大 | 状态说明、公告、风险提示 |
| 操作卡 | 带图标托盘与 CTA | 快捷功能入口 |
| 列表卡 | 纯白底 + 分隔线 | 资产列表、收益列表、聪明钱动态 |
| 消息卡 | 卡片化气泡或系统回复块 | 对话页 AI 回应、策略摘要 |

### 9.4 导航组件

导航组件分为底部主导航、页面头部导航、分段切换三类。底部主导航为全局固定骨架；页面头部导航必须服务于当前页主要任务，不得为了“看起来丰富”而堆砌功能入口；分段切换只应用于语义明确的二分结构，例如“对话 / 社区”。

| 导航类型 | 规范 | 禁止事项 |
|---|---|---|
| 底部主导航 | 五栏固定，白底浮层，选中紫色 | 过多隐藏级入口混入其中 |
| 页面头部 | 简洁、单任务导向 | 左右两边都堆按钮 |
| 分段切换 | 浅紫底托 + 激活胶囊 | 用普通文字假装切换控件 |

### 9.5 列表与 Cell

列表页中的单元格必须遵守“左识别、中信息、右结果”的结构原则。左侧负责图标或头像识别，中间是标题与辅助信息，右侧是数值、状态或箭头。这样可确保资产列表、收益列表、搜索结果、地址结果和动态列表都能共享统一骨架。

| 区域 | 内容 | 规范 |
|---|---|---|
| 左侧 | 图标 / 头像 / Token Logo | 尺寸统一，置于圆角底盘中 |
| 中间 | 主标题 + 辅助说明 | 标题 15-17，辅助文案 12-13 |
| 右侧 | 金额 / 状态 / 箭头 | 右对齐，数值优先 |

### 9.6 标签、Toast 与状态反馈

标签必须采用轻底色 + 强语义色的表达方式，而不是纯色大块。Toast / Snackbar 用于轻量反馈，例如“验证码已发送”“已复制地址”“策略已同步”；重要风险必须使用独立信息卡或弹层，而不能仅用 Toast 一闪而过。

| 类型 | 形式 | 场景 |
|---|---|---|
| 状态标签 | 浅底 + 功能色文字 | 运行中、待处理、成功 |
| Toast / Snackbar | 底部浮层、短时显示 | 成功、轻提醒、复制反馈 |
| 错误提示 | 输入下方文案 + 边框变色 | 表单校验失败 |
| 风险卡 | 独立卡片、常驻直到用户处理 | 高风险操作、收益异常 |

## 十、交互规范

H Wallet 的交互设计目标是“**自然、轻快、可预期**”。动效不是装饰，而是状态变化的解释器。页面切换、列表刷新、消息送达、按钮按压、表单错误、策略同步等关键动作都应具备一致的反馈逻辑。用户在执行金融相关操作时，需要知道系统“是否收到、是否处理中、是否成功、若失败原因是什么”。

### 10.1 动效节奏

| 动效类型 | 时长建议 | 缓动 | 说明 |
|---|---:|---|---|
| 按钮按压 | 100-140ms | ease-out | 轻缩放或透明度变化 |
| 卡片进入 | 180-240ms | ease-out | 列表初次加载、模块切换 |
| 页面转场 | 220-280ms | standard | 保持原生感，不做夸张飞入 |
| Toast 出现 | 180ms | ease-out | 快速但不突兀 |
| 骨架切换为真实内容 | 160-220ms | crossfade | 降低闪烁感 |

### 10.2 手势与操作反馈

| 场景 | 规范 |
|---|---|
| 列表刷新 | 下拉刷新使用品牌紫色指示器，不可过度花哨 |
| 卡片点击 | 整卡可点击时必须整卡响应，不可只有局部文字可点 |
| 主要按钮 | 点击后 120ms 内必须出现视觉反馈 |
| 链上操作 | 必须展示“准备中 / 等待签名 / 已提交 / 已完成 / 失败”完整状态链 |
| 搜索结果 | 输入后 300ms debounce，可显示加载态与空态 |

### 10.3 可访问性要求

React Native Paper 特别强调交互、动画和可访问性的重要性[4]。因此，H Wallet 在视觉和交互规范中必须显式加入以下要求：文本与背景对比度要满足基础可读性；所有图标按钮必须有可理解的可点击面积；输入错误不能只靠颜色区分；搜索框、发送按钮、返回按钮需提供明确语义标签。

| 规范项 | 最低要求 |
|---|---|
| 点击热区 | 不小于 44 × 44 |
| 文本对比 | 主文本与背景需满足清晰可读 |
| 错误提示 | 颜色 + 文案双重表达 |
| 加载反馈 | 不得无反馈等待超过 400ms |
| 图标语义 | 重要图标需配合文字或无障碍标签 |

## 十一、页面级布局规范

以下页面规范为本次文档的核心执行部分。所有后续页面实现与改版，均以本节为准。

### 11.1 登录页

登录页是品牌首屏，也是用户第一次感受到 H Wallet 气质的入口。页面顶部必须保留品牌 Logo、品牌名和副标题“对话式 Web3 钱包”，中部为主表单卡，底部为三张卖点卡片。新版规范明确要求：**取消密码输入框**，整个登录流程只保留邮箱输入、验证码输入以及“发送验证码”动作。

新版登录页的主表单卡结构应为：顶部登录 / 注册分段切换；其下为邮箱输入框；再下为验证码输入框与发送验证码按钮的组合；底部为主按钮“继续进入”。状态提示区需放在验证码输入框与主按钮之间，用于展示发送成功、验证码错误、登录中等状态。

| 区域 | 规范 |
|---|---|
| 顶部品牌区 | Logo 在左或居中均可，但品牌露出必须简洁清晰 |
| 表单结构 | 登录/注册切换 + 邮箱 + 验证码 + 主按钮 |
| 已删除项 | 密码输入框，不再出现 |
| 辅助信息 | 状态说明、验证码发送结果、协议文案可选 |
| 底部卖点区 | AI 对话钱包 / Agent 自动任务 / 链上资产总览 |

### 11.2 对话页（默认首页）

对话页是默认首页，必须体现“进入 App 就能开口提需求”的产品策略。因此，**顶部的‘钱包入口’和‘我的’按钮全部移除**。头部只保留页面必要识别信息，推荐两种实现方式：其一是仅保留居中的品牌/页面标识；其二是保留更轻量的“对话 / 社区”切换控件，但左右不得再挂载额外按钮。

对话页下方依次为快捷能力卡、消息流、底部输入区。快捷能力卡应保持 2×2 或横滑结构，优先展示“AI 代币分析”“一句话转账”“智能赚币推荐”“社区策略追踪”等高频任务。消息流采用左侧 AI 卡片、右侧用户消息的清晰分区表达；输入区固定底部，右侧使用品牌紫发送按钮。

| 区域 | 新规范 |
|---|---|
| 头部左侧 | 删除“钱包入口” |
| 头部右侧 | 删除“我的” |
| 头部中心 | 仅保留必要标题或轻量切换 |
| 消息区 | 卡片化 AI 回复 + 右侧用户气泡 |
| 输入区 | 大圆角白色底板 + 紫色发送按钮 |

### 11.3 钱包页

钱包页的角色是“资产底座与操作状态页”。它需要承接用户从 AI 对话流中跳转过来的资产查看、收益确认、地址复制与动作回查等行为。因此，钱包页头部必须体现明确的回流路径。新版规范要求：**右上角铃铛图标改为返回键，返回对话页**。若实现为左侧返回更符合系统习惯，则页面上仍需保证“返回对话页”是首要头部动作，不得再出现消息铃铛。

钱包页的主体结构仍由总资产 Hero 卡、快捷操作区、资产分布 / 地址信息、资产列表构成。Hero 卡以渐变为主，列表与地址模块以白色卡片为主。钱包页不应承担过多运营性内容，必须首先服务“资产可读、地址可复制、结果可确认”。

| 区域 | 规范 |
|---|---|
| 头部右侧 | 返回按钮，目标页为对话页 |
| 头部禁止项 | 铃铛、消息提醒类入口 |
| 首屏模块 | 总资产 Hero 卡 |
| 中段模块 | 快捷操作、地址与链信息 |
| 底部模块 | 资产列表、最近活动或说明 |

### 11.4 行情页

行情页用于承接“查价格、看涨跌、判断机会”的任务。页面风格应延续统一的浅色卡片结构，不应变成纯交易所风格。首屏可以由市场概览、热门币种、用户关注列表三部分构成，涨跌颜色需要明确但控制饱和度。重要信息必须一眼可扫读，避免密集表格感。

| 模块 | 规范 |
|---|---|
| 顶部摘要 | 今日市场概览或关注列表 |
| 主列表 | Token Logo + 名称 + 价格 + 涨跌 + 变化趋势 |
| 颜色表达 | 上涨绿色、下跌红色，但不使用荧光色 |
| 辅助元素 | 可使用极简折线图，不喧宾夺主 |

### 11.5 赚币页

赚币页是 H Wallet 的第二增长主线，其设计目标是让用户在理解风险与收益的前提下，更快做出申购、跟踪与调整动作。页面结构建议依次为：AI 推荐区、筛选区、产品列表、我的持仓收益、聪明钱动态。这里的“AI 推荐区”必须在视觉上具备首屏主导地位，可采用 Hero 卡或大卡片形式承载。

赚币页要避免做成纯表格式 DeFi 浏览器。产品列表虽然可以展示 APY、TVL、链、协议等字段，但推荐理由、风险标签、可申购动作与收益跟踪入口必须被设计成**可操作的产品卡片**，而不是仅供观看的数据块。

| 模块 | 规范 |
|---|---|
| AI 推荐区 | 首页首屏重点模块，展示推荐理由、风险偏好与操作入口 |
| 筛选区 | 稳健 / 平衡 / 激进，稳定币 / 主流币等轻量筛选 |
| 产品列表 | 卡片化展示，不做复杂表格 |
| 我的收益 | 已投产品、今日收益、累计收益、APR |
| 聪明钱动态 | 以简洁列表形式展示地址、动作、协议与金额 |

### 11.6 社区页

社区页在新版规范中被重新定义为“**搜索驱动的发现页**”。因此，顶部不再使用“社区 / Agent 中枢”标题区域，而必须改为搜索框。搜索框是社区页首要视觉焦点，支持搜索**代币、合约、地址**，并配合搜索历史、热门推荐或最近查看结果。

社区页不再以“控制台式任务面板”作为首屏主模块，而应改为“搜索框 + 热门发现 / 智能推荐 / 最近动态”的组合结构。若需要继续保留 Agent 任务内容，建议下沉为二级模块，而不是占据头部主语义。

| 区域 | 新规范 |
|---|---|
| 顶部主控件 | 搜索框 |
| 搜索范围 | 代币、合约、地址 |
| 旧标题区 | 删除“社区 / Agent 中枢”标题结构 |
| 首屏内容 | 热门搜索、最近浏览、推荐观察对象 |
| 二级内容 | 策略动态、地址跟踪、Agent 相关模块 |

### 11.7 我的页

“我的”页面承担账户与偏好设置职责，应保持安静、清晰、低干扰。页面需突出账户信息、风险偏好、主题模式、帮助与设置项。与其他页面相比，“我的”页不适合使用大面积渐变 Hero 卡，建议更多采用白色信息卡与列表式设置结构。

| 模块 | 规范 |
|---|---|
| 账户区 | 邮箱、地址摘要、身份状态 |
| 偏好区 | 风险偏好、通知偏好、主题模式 |
| 帮助区 | 帮助中心、反馈、关于我们 |
| 视觉风格 | 更克制，少动效，重信息结构 |

## 十二、亮色 / 暗色模式规范

H Wallet 当前主基调是亮色模式，且品牌优势也主要建立在暖白与柔紫体系之上。因此，**亮色模式为默认模式，暗色模式为增强模式**。暗色模式不能简单把白底改黑底，而必须保留 H Wallet 的“轻科技、柔和金融感”，避免退回传统 Web3 的高压黑紫霓虹视觉。

### 12.1 亮色模式

亮色模式以暖白、浅紫和白色玻璃卡为主。正文阅读、输入、表单、资产列表都应默认在亮色模式下达到最佳可读性。亮色模式是演示、汇报、市场传播和默认用户路径的首选。

### 12.2 暗色模式

暗色模式建议采用**深蓝黑底 + 低亮度紫色高亮 + 半透明深色卡片**。品牌主色仍然保留紫色，但亮度应下降，避免刺眼。暗色模式下的错误色、成功色和预警色都应同步调暗，并提高正文对比度。

| 语义角色 | 亮色模式 | 暗色模式 |
|---|---|---|
| 页面背景 | `#FCFAFF` | `#0F1020` |
| 分区背景 | `#F7F4FF` | `#17192B` |
| 卡片背景 | `rgba(255,255,255,0.88)` | `rgba(28,30,49,0.88)` |
| 主文本 | `#1A1A2E` | `#F5F7FF` |
| 次文本 | `#666C85` | `#B6BCD3` |
| 主色 | `#7C3AED` | `#9E7BFF` |
| 分隔线 | `#E8EAF2` | `rgba(255,255,255,0.10)` |

### 12.3 主题切换原则

主题切换必须基于语义 token 统一切换，不允许在页面中直接硬编码两套颜色。当前仓库已有 `theme.ts` 结构，后续需进一步把设计 token 与运行时主题收敛为统一命名体系，例如 `text.primary`、`text.secondary`、`surface.card`、`brand.primary` 等，而不是继续扩大页面局部常量规模[3]。

## 十三、实现约束与落地建议

为了让设计规范真正能被执行，本节给出与工程实现直接相关的约束。首先，颜色、间距、圆角、阴影、字体不应再分散写在页面文件中，应逐步收敛到主题或常量层。其次，头部组件、输入组件、主按钮与卡片容器需要形成复用抽象，否则老板的页面修改会在多个文件中重复返工。

此外，当前主题层存在 token 语义不完全统一的情况，例如运行时色板字段与页面局部命名并未完全对齐。新版设计规范已经给出明确语义命名建议，后续前端需据此完成收敛，减少“同一颜色有多个名字、同一组件多个实现”的维护成本。

| 实施优先级 | 内容 | 目标 |
|---|---|---|
| P0 | 覆盖登录页、对话页、钱包页、社区页 4 项老板要求 | 立即对齐业务期望 |
| P0 | 收敛颜色、字体、间距、圆角 token | 建立设计系统底座 |
| P1 | 抽象按钮、输入框、卡片、搜索框、头部组件 | 降低后续改版成本 |
| P1 | 对赚币页、行情页、我的页进行统一风格升级 | 完成产品一致性 |
| P2 | 完整补齐暗色模式、可访问性、微交互细节 | 提升产品完成度 |

## 十四、最终设计决策摘要

H Wallet V2 的核心设计决策可以概括为一句话：**用更轻的视觉，承载更强的链上能力，用 AI 对话作为默认入口，把钱包、赚币、搜索和社区都组织成可执行的移动端体验。** 这也是本版设计规范替换旧版文档的根本原因。未来所有页面设计与开发评审，都应首先检查是否符合以下四条标准：是否足够移动端、是否足够聚焦、是否足够可信、是否足够一致。

| 决策维度 | 最终结论 |
|---|---|
| 默认首页 | 对话页 |
| 核心品牌风格 | 亮色、暖白、柔紫、轻玻璃 |
| 登录方式 | 邮箱 + 验证码 |
| 钱包头部 | 返回对话页，不保留铃铛 |
| 社区定位 | 搜索驱动的发现页 |
| 设计系统组织方式 | 语义 token + 组件规范 + 页面规范 |
